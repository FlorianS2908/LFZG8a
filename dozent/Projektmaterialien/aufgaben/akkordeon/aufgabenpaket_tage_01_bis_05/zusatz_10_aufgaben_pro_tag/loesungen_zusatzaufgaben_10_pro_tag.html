Hinweis zur Nutzung

Die Lösungen sind als Musterlösungen zu verstehen. Bei HTML/CSS gibt es oft mehrere fachlich richtige Wege. Entscheidend sind Semantik, Wartbarkeit, Responsiveness, Bedienbarkeit und nachvollziehbare CSS-Struktur.

Tag 1

Lösungen zu Tag 1: HTML-Grundstruktur und CSS-Basics

Fundament für sauberes HTML, sichtbare Grundstruktur, CSS-Basisregeln, Box-Modell, Farben, Abstände und einfache Selektoren.

Aufgaben Tag 1
Normal: Akkordeon-Seite mit sauberem HTML-Grundgerüst anlegen

Ziel: Die Teilnehmenden erstellen eine vollständige HTML-Datei für das Akkordeon und erklären die Rolle von head, body, title und meta.

  • Ein vollständiges Dokument besitzt genau eine Wurzelstruktur mit `html`, einen technischen `head` und sichtbare Inhalte im `body`.
  • Der Viewport-Meta-Tag ist Pflicht, damit später responsive CSS-Regeln sinnvoll wirken.
  • Die Überschrift gehört nicht in den `head`, sondern sichtbar in den `body`.

Muster / Codeauszug:

<!doctype html>
<html lang="de">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Akkordeon Tag 1</title>
  <link rel="stylesheet" href="style.css">
</head>
<body>
  <header class="site-header">
    <h1>CSS-Akkordeon</h1>
  </header>
  <main class="accordion-shell">
    <p>Start des Akkordeon-Projekts.</p>
  </main>
</body>
</html>

Prüfung: Browser zeigt Titel, Überschrift und Hauptbereich; DevTools zeigen korrekte Grundstruktur.

Normal: Semantische Seitenbereiche für das Akkordeon strukturieren

Ziel: Die Akkordeon-Seite wird mit header, main, section, article und footer logisch gegliedert.

  • `section` beschreibt den thematischen Bereich „Akkordeon“.
  • `article` eignet sich für einzelne, eigenständig verständliche Akkordeon-Einträge.
  • `div` wird nur benötigt, wenn kein semantisches Element passt.

Muster / Codeauszug:

<main>
  <section class="accordion" aria-labelledby="acc-title">
    <h2 id="acc-title">Akkordeon-Inhalte</h2>
    <article class="accordion-item">
      <h3>Panel 1</h3>
      <p>Kurzer Inhalt zum ersten Panel.</p>
    </article>
  </section>
</main>

Prüfung: Im HTML sind die Seitenbereiche fachlich sinnvoll erkennbar und nicht nur aus div-Containern aufgebaut.

Normal: CSS-Basis mit Farben, Schrift und Abständen erstellen

Ziel: Die Teilnehmenden setzen Grundregeln für body, Überschriften, Absätze und den Akkordeon-Container.

  • Globale Regeln gehören an `body`; komponentenbezogene Regeln an Klassen wie `.accordion-shell`.
  • `box-sizing:border-box` erleichtert spätere Breitenberechnung.
  • Padding erzeugt Innenabstand, margin Außenabstand, border einen sichtbaren Rahmen.

Muster / Codeauszug:

* { box-sizing: border-box; }
body {
  margin: 0;
  font-family: Arial, sans-serif;
  color: #173248;
  background: #eef8fb;
}
.accordion-shell {
  max-width: 960px;
  margin: 2rem auto;
  padding: 1rem;
  background: #fff;
  border: 1px solid #d8e8ee;
}

Prüfung: Seite wirkt sichtbar gestaltet; Abstände sind bewusst gesetzt und nicht zufällig vom Browserstandard abhängig.

Normal: Listen, Links und Bildplatzhalter im Akkordeon sinnvoll verwenden

Ziel: Ein Akkordeon-Eintrag bekommt einen Text, eine ungeordnete Liste, einen Link und einen Bildplatzhalter.

  • Listen ersetzen keine Absätze mit Bindestrichen; sie geben Struktur.
  • Links brauchen aussagekräftige Linktexte.
  • Bilder brauchen `alt`; bei einem reinen CSS-Platzhalter kann eine Beschriftung im Text reichen.

Muster / Codeauszug:

<article class="accordion-item" id="panel-html">
  <h3>HTML-Grundlagen</h3>
  <p>Dieses Panel zeigt wichtige Bestandteile.</p>
  <ul>
    <li>Überschrift</li>
    <li>Absatz</li>
    <li>Liste</li>
  </ul>
  <a href="#panel-css">Zum CSS-Panel</a>
</article>

Prüfung: Die Inhalte sind semantisch passend: Liste als Liste, Link als Link, Bild/Platzhalter mit Beschreibung.

Normal: Klassen und IDs fachlich sauber unterscheiden

Ziel: Die Teilnehmenden benennen HTML-Elemente so, dass später CSS eindeutig und wartbar bleibt.

  • Klassen sind für wiederholbare Gestaltung gedacht.
  • IDs sind eindeutig und eignen sich für Anker, `for`/`id`-Verknüpfungen oder gezielte Sprungmarken.
  • Für fünf Panels ist `.accordion-item` besser als fünf komplett unterschiedliche Klassen.

Muster / Codeauszug:

<section class="accordion" id="akkordeon-start">
  <article class="accordion-item" id="panel-1">...</article>
  <article class="accordion-item" id="panel-2">...</article>
</section>

Prüfung: Die Akkordeon-Items verwenden wiederholbare Klassen; IDs sind eindeutig und nicht mehrfach vorhanden.

Schwer: Divitis im Akkordeon reduzieren und semantisch refactoren

Ziel: Eine absichtlich div-lastige Akkordeon-Struktur wird in semantische HTML-Elemente überführt.

  • Semantik verbessert Lesbarkeit, Barrierefreiheit und Wartbarkeit.
  • Nicht jeder Container ist falsch, aber fachliche Inhalte sollten passende HTML-Elemente bekommen.
  • CSS-Klassen können nach dem Refactoring meist erhalten bleiben.

Muster / Codeauszug:

<!-- vorher: <div class="item"> -->
<article class="accordion-item">
  <h3>Semantisches Panel</h3>
  <p>Inhalt bleibt erhalten, Bedeutung wird klarer.</p>
</article>

Prüfung: Die Seite sieht gleich oder besser aus, hat aber eine fachlich klarere HTML-Struktur.

Schwer: CSS-Variablen für das Akkordeon-Farbsystem einführen

Ziel: Farben und Abstände werden über Custom Properties zentral steuerbar gemacht.

  • Custom Properties eignen sich für wiederkehrende Werte wie Farben, Abstände, Radius und Schatten.
  • Sie erhöhen Konsistenz und reduzieren Sucharbeit im CSS.
  • Variablen dürfen sprechend benannt werden, nicht nur `--blue1`.

Muster / Codeauszug:

:root {
  --brand: #003964;
  --accent: #00abc7;
  --surface: #ffffff;
  --line: #d8e8ee;
  --space: 1rem;
}
.accordion-item {
  background: var(--surface);
  border: 1px solid var(--line);
  padding: var(--space);
}

Prüfung: Mehrere Komponenten nutzen dieselben Variablen; Farbanpassung ist zentral möglich.

Schwer: Spezifitätsfehler im Start-CSS finden und beheben

Ziel: Die Teilnehmenden analysieren, warum bestimmte CSS-Regeln nicht greifen, und lösen Konflikte nachvollziehbar.

  • `!important` ist in Unterrichtsprojekten meist ein Warnsignal.
  • Spezifität steigt durch Klassen, IDs und Kontextselektoren.
  • Besser ist eine klare Komponentenklasse mit kontrollierter Reihenfolge.

Muster / Codeauszug:

/* problematisch */
h3 { color: red; }
.accordion-item h3 { color: #003964; }

/* sauber */
.accordion-title { color: var(--brand); }

Prüfung: Konflikt ist behoben; Erklärung nennt mindestens zwei Ursachen: Reihenfolge, Spezifität oder falscher Selektor.

Schwer: HTML-Kommentare als technische Dokumentation einsetzen

Ziel: Der Startstand des Akkordeons wird so kommentiert, dass ein anderer Entwickler die Struktur versteht.

  • Gute Kommentare erklären das Warum, nicht das Offensichtliche.
  • Bei HTML/CSS-Projekten sind Kommentare für Strukturentscheidungen und Abhängigkeiten sinnvoll.
  • Kommentare dürfen später im Produktivcode reduziert werden, sind im Lernprojekt aber wertvoll.

Muster / Codeauszug:

<!-- Akkordeon-Item: Titel und Inhalt bleiben zusammen, damit die Komponente später kopierbar bleibt. -->
<article class="accordion-item">
  <h3 class="accordion-title">Panel 1</h3>
  <p class="accordion-content">Inhalt...</p>
</article>

Prüfung: Kommentare helfen beim Verständnis, ohne den Code zu überladen.

Schwer: Mini-Abnahme für die HTML- und CSS-Basis durchführen

Ziel: Die Teilnehmenden prüfen ihren Tag-1-Stand systematisch anhand technischer Kriterien.

  • Eine Abnahme ist fachlich stärker als „sieht gut aus“.
  • Prüfkriterien sollten beobachtbar sein.
  • TODO-Kommentare helfen, offene Punkte später gezielt zu bearbeiten.

Muster / Codeauszug:

<!-- TODO: Alt-Text ergänzen -->
<!-- TODO: Klasse vereinheitlichen: acc-item statt panel-box -->
<!-- TODO: Feste Breite durch max-width ersetzen -->

Prüfung: Checkliste liegt vor; mindestens drei konkrete Korrekturen wurden umgesetzt.

Tag 2

Lösungen zu Tag 2: Navigation, Buttons und Flexbox

Navigation, klickbare Elemente, Button-/Link-Unterscheidung und Flexbox für Bedienleisten, Panel-Köpfe und Komponentenlayout.

Aufgaben Tag 2
Normal: Akkordeon-Navigation mit nav und Linkliste bauen

Ziel: Die Teilnehmenden erstellen eine interne Navigation zu den Akkordeon-Panels.

  • `nav` ist für zentrale Navigation sinnvoll.
  • Eine Linkliste bleibt auch dann eine Liste, wenn sie optisch als Buttonleiste aussieht.
  • IDs müssen zu den href-Ankern passen.

Muster / Codeauszug:

<nav class="panel-nav" aria-label="Akkordeon-Panels">
  <ul>
    <li><a href="#panel-1">Panel 1</a></li>
  </ul>
</nav>

.panel-nav ul { display:flex; gap:.5rem; flex-wrap:wrap; }

Prüfung: Navigation ist semantisch als nav umgesetzt und alle Anker funktionieren.

Normal: Panel-Kopf mit Flexbox ausrichten

Ziel: Titel, Nummer und Icon eines Akkordeon-Panels werden in einer Zeile flexibel ausgerichtet.

  • Flexbox eignet sich für eine horizontale Ausrichtung innerhalb einer Komponente.
  • `gap` ist sauberer als viele Einzel-margins.
  • Nummer und Icon sollten nicht die Überschrift verdrängen.

Muster / Codeauszug:

.accordion-head {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: .75rem;
}

Prüfung: Panel-Kopf bleibt übersichtlich und bricht sauber um oder bleibt lesbar.

Normal: Link und Button fachlich unterscheiden

Ziel: Die Teilnehmenden entscheiden, wann ein a-Element und wann ein button-Element korrekt ist.

  • Ein Link navigiert zu einem Ziel.
  • Ein Button löst eine Aktion aus.
  • Im CSS können beide ähnlich aussehen, ihre Bedeutung bleibt aber unterschiedlich.

Muster / Codeauszug:

<a class="btn-link" href="#panel-2">Zum Panel</a>
<button class="btn" type="button">Ansicht prüfen</button>

Prüfung: Navigation nutzt a; Aktionen nutzen button. Keine Buttons werden als reine Sprunglinks missbraucht.

Normal: Hover- und Fokuszustände für Akkordeon-Bedienung gestalten

Ziel: Klickbare Elemente erhalten sichtbare Zustände für Maus- und Tastaturbedienung.

  • Hover allein reicht nicht, weil Tastatur- und Touch-Nutzung anders funktionieren.
  • `outline` sollte nicht ersatzlos entfernt werden.
  • `:focus-visible` ist für moderne Tastaturfokusse gut geeignet.

Muster / Codeauszug:

.panel-nav a:hover { background:#003964; color:#fff; }
.panel-nav a:focus-visible, button:focus-visible {
  outline: 3px solid #00abc7;
  outline-offset: 3px;
}

Prüfung: Aktive Bedienflächen sind mit Maus und Tastatur klar erkennbar.

Normal: Flex-Wrap für eine stabile Panel-Leiste nutzen

Ziel: Eine horizontale Leiste mit Panel-Links bricht bei wenig Platz sauber um.

  • `flex-wrap` verhindert Überlauf, wenn Elemente zu breit werden.
  • `gap` arbeitet in Zeilen und Spalten gleichmäßig.
  • Feste Breiten sind für Navigationschips meist schlechter als flexible Inhalte.

Muster / Codeauszug:

.panel-nav ul {
  display:flex;
  flex-wrap:wrap;
  gap:.5rem;
  padding:0;
  list-style:none;
}

Prüfung: Die Leiste verursacht keinen horizontalen Scrollbalken.

Schwer: CSS-only Bedienzustand mit Radio und Label vorbereiten

Ziel: Die spätere Akkordeon-Interaktion wird mit input type=radio und label strukturell vorbereitet.

  • Radio-Buttons ermöglichen einen exklusiven Zustand: genau ein Panel aktiv.
  • Das label ist die klickbare Oberfläche.
  • Inputs dürfen nicht mit `display:none` versteckt werden, wenn Fokus/Bedienbarkeit erhalten bleiben soll.

Muster / Codeauszug:

<input class="acc-toggle" type="radio" name="accordion" id="acc-1" checked>
<label class="acc-label" for="acc-1">Panel 1</label>

.acc-toggle { position:absolute; opacity:0; }

Prüfung: Labels steuern die passenden Radio-Inputs; IDs und for-Attribute stimmen eindeutig überein.

Schwer: Komponenten-CSS für Panel-Köpfe refactoren

Ziel: Mehrfach vorhandene Panel-Kopf-Regeln werden in eine wiederverwendbare Komponentenklasse überführt.

  • Komponentenklassen beschreiben Rollen innerhalb der Komponente.
  • Wiederverwendbare CSS-Struktur macht spätere Änderungen sicherer.
  • Spezielle Abweichungen werden als Modifier ergänzt, nicht als komplette Kopie.

Muster / Codeauszug:

.accordion-head { display:flex; align-items:center; gap:.75rem; }
.accordion-title { flex:1; margin:0; }
.accordion-icon { font-weight:900; }

Prüfung: Doppelte CSS-Regeln sind reduziert; Komponentenklassen sind sprechend benannt.

Schwer: Flexbox-Fehleranalyse mit DevTools durchführen

Ziel: Die Teilnehmenden finden und beheben ein Layoutproblem in einer Flexbox-Leiste.

  • Flex-Items können durch lange Texte oder feste Breiten überlaufen.
  • `min-width:0` ist oft nötig, damit Text in Flex-Items schrumpfen darf.
  • Flexbox löst Ausrichtung, aber nicht automatisch jede Inhaltsbreite.

Muster / Codeauszug:

.accordion-head > .accordion-title {
  flex: 1 1 auto;
  min-width: 0;
}

Prüfung: Kein horizontaler Überlauf; Ursache ist schriftlich nachvollziehbar.

Schwer: Accessible Name für klickbare Akkordeon-Elemente prüfen

Ziel: Klickbare Labels, Links und Buttons bekommen verständliche sichtbare Texte und bei Bedarf ARIA-Ergänzungen.

  • Der sichtbare Text sollte Ziel oder Aktion beschreiben.
  • Icon-only Buttons brauchen einen zugänglichen Namen.
  • ARIA ergänzt Bedeutung, ersetzt aber keine saubere HTML-Struktur.

Muster / Codeauszug:

<button type="button" aria-label="Panel 2 öffnen">+</button>
<a href="#panel-3">Zum Akkordeon-Panel „CSS Grid“</a>

Prüfung: Alle interaktiven Elemente sind ohne Kontext verständlich beschriftet.

Schwer: Zustandsdesign für aktiv, hover und focus entwerfen

Ziel: Die Teilnehmenden definieren ein konsistentes Zustandsdesign für alle Akkordeon-Steuerelemente.

  • Aktivzustände sollten mit Farbe plus Form, Rahmen oder Icon erkennbar sein.
  • Fokuszustände dürfen nicht von Hoverzuständen überdeckt werden.
  • Konsistenz über alle Steuerelemente reduziert Bedienfehler.

Muster / Codeauszug:

.acc-label { border:1px solid var(--line); }
.acc-label:hover { background:var(--soft); }
.acc-label:focus-visible { outline:3px solid var(--accent); }
.acc-toggle:checked + .acc-label { border-left:6px solid var(--accent); }

Prüfung: Zustände sind visuell unterscheidbar und barriereärmer umgesetzt.

Tag 3

Lösungen zu Tag 3: Struktur, Karten und Grid

Struktur, Karten, Grid, figure/figcaption, aside und mehrspaltige Akkordeon-Layouts.

Aufgaben Tag 3
Normal: Akkordeon-Panels als Karten gestalten

Ziel: Jedes Akkordeon-Panel erhält Kartenoptik mit Rahmen, Radius, Schatten und Innenabstand.

  • Kartenoptik entsteht durch Hintergrund, Abstand, Rahmen, Radius und Schatten.
  • Zu starke Schatten wirken schnell unprofessionell.
  • Einheitliche Klassen vermeiden Stilbrüche.

Muster / Codeauszug:

.accordion-card {
  padding: 1rem;
  background: #fff;
  border: 1px solid #d8e8ee;
  border-radius: .75rem;
  box-shadow: 0 10px 24px rgba(0,57,100,.08);
}

Prüfung: Alle Panels wirken als einheitliche Karten, ohne unruhige Einzelgestaltung.

Normal: Kartenübersicht mit CSS Grid erstellen

Ziel: Mehrere Zusatzkarten zum Akkordeon werden mit Grid in Spalten angeordnet.

  • Grid eignet sich für zweidimensionale Raster.
  • `gap` ist für Rasterabstände übersichtlich.
  • Für starre drei Spalten ist `repeat(3, 1fr)` geeignet.

Muster / Codeauszug:

.info-grid {
  display: grid;
  grid-template-columns: repeat(3, 1fr);
  gap: 1rem;
}

Prüfung: Infokarten stehen auf Desktop nebeneinander und haben gleichmäßige Abstände.

Normal: Aside als Zusatzbereich zum Akkordeon einbauen

Ziel: Ein ergänzender Hinweisbereich wird semantisch mit aside umgesetzt.

  • `aside` ist für ergänzende Inhalte geeignet.
  • Die Infobox sollte den Hauptinhalt unterstützen, nicht ersetzen.
  • Ein border-left eignet sich gut für Hinweischarakter.

Muster / Codeauszug:

<aside class="help-box">
  <h2>Hinweis</h2>
  <p>Wähle ein Panel aus, um den Inhalt zu öffnen.</p>
</aside>

Prüfung: aside enthält Zusatzinformationen und ist optisch vom Hauptinhalt getrennt.

Normal: Figure und Figcaption für Akkordeon-Bildmaterial verwenden

Ziel: Ein Bild oder Platzhalter im Akkordeon wird mit figure und figcaption fachlich korrekt eingebunden.

  • `figure` gruppiert Medien und Beschriftung.
  • `figcaption` beschreibt das Bild oder seinen Zweck.
  • Bei echten Bildern bleibt `alt` trotzdem wichtig.

Muster / Codeauszug:

<figure class="media-card">
  <img src="bild.jpg" alt="Akkordeon-Demo">
  <figcaption>Beispiel für ein Panelbild.</figcaption>
</figure>

Prüfung: Bild und Beschreibung sind semantisch gekoppelt.

Normal: Footer und Header im Kartenlayout vereinheitlichen

Ziel: Panelkarten erhalten klare Kopf- und Fußbereiche.

  • Header/Footer können auch innerhalb von article sinnvoll sein.
  • Eine Karte kann intern ebenfalls strukturiert sein.
  • Gleiche Struktur erleichtert CSS und spätere Pflege.

Muster / Codeauszug:

<article class="accordion-card">
  <header><h3>Panel</h3></header>
  <p>Inhalt</p>
  <footer>Stand: Tag 3</footer>
</article>

Prüfung: Karten haben ein konsistentes Innenlayout mit wiedererkennbaren Bereichen.

Schwer: Auto-fit Grid für variable Kartenanzahl umsetzen

Ziel: Die Kartenübersicht passt sich automatisch an verfügbare Breite und Anzahl der Karten an.

  • `auto-fit` und `minmax` machen Grid besonders flexibel.
  • Eine Mindestbreite verhindert unlesbare Karten.
  • `1fr` verteilt verfügbaren Restplatz gleichmäßig.

Muster / Codeauszug:

.info-grid {
  display:grid;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  gap:1rem;
}

Prüfung: Kartenraster reagiert flexibel und benötigt keine zusätzliche Media Query für jede Breite.

Schwer: Grid-Template-Areas für Seitenlayout bauen

Ziel: Akkordeon, aside und Zusatzkarten werden mit benannten Grid-Bereichen angeordnet.

  • Grid-Areas machen komplexere Layouts verständlich.
  • Die HTML-Reihenfolge sollte weiterhin logisch bleiben.
  • CSS entscheidet Darstellung, HTML bleibt semantisch.

Muster / Codeauszug:

.page-layout {
  display:grid;
  grid-template-columns: 2fr 1fr;
  grid-template-areas: "intro intro" "accordion aside";
  gap:1rem;
}
.intro{grid-area:intro}.accordion{grid-area:accordion}.help-box{grid-area:aside}

Prüfung: Layout nutzt benannte Bereiche und ist im CSS verständlich lesbar.

Schwer: Kartenhöhe und Inhaltsüberlauf kontrollieren

Ziel: Unterschiedlich lange Panelinhalte werden so behandelt, dass das Kartenlayout stabil bleibt.

  • Nicht jeder Inhalt sollte hart begrenzt werden.
  • Für Kartenübersichten kann eine Mindesthöhe helfen.
  • Bei Begrenzung muss Scroll- oder „Mehr lesen“-Logik klar sein.

Muster / Codeauszug:

.accordion-card {
  display:grid;
  grid-template-rows:auto 1fr auto;
  min-height: 220px;
}

Prüfung: Lange Inhalte zerstören das Layout nicht; Entscheidung ist nachvollziehbar.

Schwer: Figure-Karten responsiv in das Grid integrieren

Ziel: Medienkarten mit figure/img/figcaption bleiben im Grid sauber und überlaufen nicht.

  • Browser geben figure standardmäßig margin; das kann Layout stören.
  • Bilder brauchen flexible Breiten.
  • Beschriftungen dürfen umbrechen und sollten lesbar bleiben.

Muster / Codeauszug:

figure { margin:0; }
.media-card img {
  display:block;
  width:100%;
  height:auto;
}
.media-card figcaption { font-size:.9rem; }

Prüfung: Medienkarten bleiben sauber im Raster; Bilder überlaufen nicht.

Schwer: Grid- und Flexbox-Einsatz fachlich begründen

Ziel: Die Teilnehmenden begründen im Codekommentar, warum sie Grid oder Flexbox verwenden.

  • Flexbox ist stark für eine Achse.
  • Grid ist stark für Zeilen und Spalten gleichzeitig.
  • Die Methode sollte nach Layoutproblem gewählt werden, nicht nach Gewohnheit.

Muster / Codeauszug:

/* Flexbox: Panelkopf richtet Titel und Icon in einer Zeile aus. */
.accordion-head { display:flex; }

/* Grid: Kartenübersicht braucht Zeilen und Spalten. */
.info-grid { display:grid; }

Prüfung: Kommentare zeigen fachliche Entscheidung: eindimensional = Flexbox, zweidimensional = Grid.

Tag 4

Lösungen zu Tag 4: Responsive HTML-Elemente

Mobile First, Viewport, responsive Bilder, flexible Navigation, Media Queries, clamp/min/max und Bedienbarkeit auf kleinen Bildschirmen.

Aufgaben Tag 4
Normal: Viewport und mobile Grunddarstellung prüfen

Ziel: Die Teilnehmenden prüfen, ob die Akkordeon-Seite auf mobilen Breiten korrekt skaliert.

  • Ohne Viewport wirkt die Seite auf Mobilgeräten oft künstlich verkleinert.
  • Mobile Prüfung ist keine Geschmacksfrage, sondern technischer Test.
  • Probleme wie Überlauf, zu kleine Schrift oder zu enge Buttons müssen konkret benannt werden.

Muster / Codeauszug:

<meta name="viewport" content="width=device-width, initial-scale=1">

Prüfung: Viewport ist vorhanden; Probleme sind beobachtbar dokumentiert.

Normal: Akkordeon-Container mit width, max-width und margin responsiv machen

Ziel: Der Hauptcontainer passt sich an Bildschirmbreiten an und bleibt auf Desktop begrenzt.

  • Feste Breiten sind ein häufiger Responsiveness-Fehler.
  • `max-width` begrenzt die Lesebreite.
  • `width:min(100%, 960px)` ist eine moderne, klare Lösung.

Muster / Codeauszug:

.accordion-shell {
  width: min(100% - 2rem, 960px);
  margin-inline: auto;
}

Prüfung: Kein horizontaler Scrollbalken; Container bleibt auf Desktop angenehm lesbar.

Normal: Bilder und Medien im Akkordeon flexibel machen

Ziel: Bilder in Panels überlaufen nicht und behalten ihr Seitenverhältnis.

  • `max-width:100%` verhindert Überlauf.
  • `height:auto` erhält das Seitenverhältnis.
  • `display:block` entfernt unerwünschte Inline-Abstände.

Muster / Codeauszug:

img {
  max-width: 100%;
  height: auto;
  display: block;
}

Prüfung: Bild passt in das Panel und verursacht keinen horizontalen Scrollbalken.

Normal: Responsive Navigation mit Media Query anpassen

Ziel: Die Panelnavigation verhält sich auf kleinen Bildschirmen anders als auf Desktop.

  • Mobile-first bedeutet: Grundzustand für kleine Screens, Erweiterung für größere.
  • Breakpoints sollten inhaltlich begründet sein, nicht zufällig.
  • Klickflächen brauchen ausreichend Höhe und Abstand.

Muster / Codeauszug:

.panel-nav ul { display:grid; gap:.5rem; }
@media (min-width:700px) {
  .panel-nav ul { display:flex; flex-wrap:wrap; }
}

Prüfung: Navigation ist mobil gut bedienbar und auf Desktop platzsparend.

Normal: Schriftgrößen mit clamp flexibel gestalten

Ziel: Überschriften reagieren auf Bildschirmbreiten, ohne zu klein oder zu groß zu werden.

  • `clamp(min, bevorzugt, max)` begrenzt flexible Werte.
  • Für große Überschriften ist clamp oft besser als viele Breakpoints.
  • Lesbarkeit bleibt wichtiger als Effekt.

Muster / Codeauszug:

h1 {
  font-size: clamp(2rem, 5vw, 4rem);
  line-height: 1.05;
}

Prüfung: Schriftgröße skaliert kontrolliert zwischen Mindest- und Maximalwert.

Schwer: Mobile-first Refactoring des gesamten Akkordeons

Ziel: Das Akkordeon-CSS wird so umgebaut, dass kleine Screens die Basis bilden und Desktop nur erweitert wird.

  • Mobile-first CSS ist meist stabiler, weil kleine Bildschirme strenger sind.
  • Desktop-Regeln sind Erweiterungen, nicht Grundvoraussetzung.
  • Überläufe müssen aktiv getestet werden.

Muster / Codeauszug:

.accordion { display:grid; gap:1rem; }
@media (min-width:900px) {
  .accordion { grid-template-columns: 1fr 1fr; }
}

Prüfung: CSS ist mobile-first lesbar aufgebaut und erzeugt keine Überläufe.

Schwer: Picture und Source für responsive Akkordeon-Bilder einsetzen

Ziel: Je nach Bildschirmbreite wird eine passende Bildquelle verwendet.

  • `picture` steuert Bildquellen, nicht die Beschriftung.
  • Das `img` bleibt Pflicht als Fallback und Träger des alt-Texts.
  • source ist selbst nicht sichtbar.

Muster / Codeauszug:

<picture>
  <source media="(min-width: 800px)" srcset="akkordeon-gross.jpg">
  <img src="akkordeon-klein.jpg" alt="Akkordeon-Panel mit Bild">
</picture>

Prüfung: picture-Struktur ist korrekt; Fallback-img bleibt vorhanden.

Schwer: CSS-only Akkordeon auf kleinen Screens gestapelt darstellen

Ziel: Das Akkordeon wechselt mobil von nebeneinanderliegenden Panels zu einer gestapelten Darstellung.

  • Absolute Positionierung kann auf mobilen Screens schnell Probleme erzeugen.
  • Eine gestapelte Darstellung ist für kleine Breiten oft verständlicher.
  • Aktive Zustände müssen auf beiden Layouts funktionieren.

Muster / Codeauszug:

.accordion-list { display:grid; gap:1rem; }
@media (min-width:900px){
  .accordion-list { grid-template-columns: repeat(5, 1fr); }
}

Prüfung: Mobil sind Panels gestapelt und bedienbar; Desktop nutzt erweitertes Layout.

Schwer: Reduced Motion für Übergänge berücksichtigen

Ziel: Animationen und Übergänge werden für Nutzer mit reduzierter Bewegung entschärft.

  • Barrierearmes Design berücksichtigt Bewegungsempfindlichkeit.
  • Transitions dürfen nicht zwingend zum Verständnis nötig sein.
  • Die Media Query wirkt nur bei entsprechender Nutzereinstellung.

Muster / Codeauszug:

@media (prefers-reduced-motion: reduce) {
  * {
    transition-duration: .01ms !important;
    animation-duration: .01ms !important;
  }
}

Prüfung: Motion-Reduktion ist umgesetzt, ohne Bedienbarkeit zu verlieren.

Schwer: Responsiveness-Testmatrix für das Akkordeon erstellen

Ziel: Die Teilnehmenden entwickeln eine kleine Testmatrix für verschiedene Gerätebreiten und Bedienformen.

  • Eine Testmatrix macht Responsiveness nachvollziehbar.
  • „Passt“ ist kein ausreichendes Testergebnis.
  • Korrekturen sollten mit Ursache und Maßnahme dokumentiert werden.

Muster / Codeauszug:

| Breite | Prüfung | Ergebnis | Maßnahme |
|---|---|---|---|
| 375px | kein horizontaler Scroll | Fehler | feste Breite entfernen |

Prüfung: Testmatrix enthält konkrete Beobachtungen und Korrekturen.

Tag 5

Lösungen zu Tag 5: Formulare, Webfonts und CSS-Struktur

Formulare, Label/Input-Zuordnung, Webfonts, CSS-Struktur, Fokuszustände, Wartbarkeit und Abschlussabnahme.

Aufgaben Tag 5
Normal: Kontaktformular unter dem Akkordeon aufbauen

Ziel: Die Akkordeon-Seite erhält ein kleines Kontakt- oder Feedbackformular mit sauberer Struktur.

  • Formulare brauchen klare Beschriftungen.
  • `for` und `id` stellen eine bedienbare Verbindung her.
  • Der Submit-Button bekommt explizit `type="submit"`.

Muster / Codeauszug:

<form class="feedback-form">
  <label for="name">Name</label>
  <input id="name" name="name" type="text">
  <button type="submit">Feedback senden</button>
</form>

Prüfung: Formularfelder haben Labels; Buttontyp ist korrekt gesetzt.

Normal: Formularfelder einheitlich gestalten

Ziel: Input, textarea und select erhalten gemeinsame CSS-Regeln.

  • Formularfelder erben Schrift nicht immer automatisch sauber.
  • Gemeinsame Regeln reduzieren Wiederholung.
  • `width:100%` funktioniert mit `box-sizing:border-box` zuverlässig.

Muster / Codeauszug:

input, textarea, select {
  width:100%;
  font: inherit;
  padding:.75rem;
  border:1px solid #d8e8ee;
  border-radius:.5rem;
}

Prüfung: Formularfelder sind optisch konsistent und passen in den Container.

Normal: Fokuszustände für Formular und Akkordeon vereinheitlichen

Ziel: Alle bedienbaren Elemente bekommen einen einheitlichen Fokuszustand.

  • Fokuszustände sind wichtig für Tastaturbedienung.
  • Outline sollte nicht deaktiviert werden.
  • Eine zentrale Regel verhindert uneinheitliche Bedienzustände.

Muster / Codeauszug:

a:focus-visible, button:focus-visible, input:focus-visible, textarea:focus-visible, select:focus-visible {
  outline: 3px solid #00abc7;
  outline-offset: 3px;
}

Prüfung: Fokus ist überall sichtbar und konsistent.

Normal: CSS-Datei in Bereiche gliedern

Ziel: Das CSS wird durch Kommentarblöcke in Grundregeln, Layout, Komponenten und Utilities strukturiert.

  • CSS-Struktur ist ein Qualitätsmerkmal.
  • Eine sinnvolle Reihenfolge reduziert Suchzeit.
  • Kommentarblöcke helfen besonders in Lernprojekten.

Muster / Codeauszug:

/* 1. Basis */
/* 2. Layout */
/* 3. Akkordeon-Komponenten */
/* 4. Formular */
/* 5. Utilities */

Prüfung: CSS ist lesbar gegliedert und neue Regeln lassen sich logisch einordnen.

Normal: Webfont mit Fallback-System vorbereiten

Ziel: Eine Schriftfamilie wird bewusst mit Fallbacks eingesetzt.

  • Fallbacks greifen, wenn die erste Schrift nicht verfügbar ist.
  • Systemschriften sind robust und schnell.
  • Webfonts sollten sparsam und bewusst genutzt werden.

Muster / Codeauszug:

body {
  font-family: "Inter", Arial, Helvetica, sans-serif;
}

Prüfung: font-family enthält mehrere sinnvolle Fallbacks und ist zentral gesetzt.

Schwer: Formularvalidierung visuell und barriereärmer darstellen

Ziel: Pflichtfelder und Fehlerzustände werden nicht nur über Farbe kommuniziert.

  • Farbe allein ist keine ausreichende Fehlermeldung.
  • Hilfetexte können über `aria-describedby` zugeordnet werden.
  • Fehlermeldungen sollten konkret sagen, was korrigiert werden muss.

Muster / Codeauszug:

<label for="email">E-Mail <span aria-hidden="true">*</span></label>
<input id="email" aria-describedby="email-help">
<p id="email-help">Bitte eine gültige E-Mail-Adresse eingeben.</p>

Prüfung: Fehler sind auch ohne Farbwahrnehmung verständlich.

Schwer: CSS-Architektur mit Komponenten und Modifiern refactoren

Ziel: Das Akkordeon-CSS wird in wiederverwendbare Komponentenklassen und Modifier aufgeteilt.

  • Komponentenklassen sind stabiler als tiefe Strukturselektoren.
  • Modifier beschreiben Varianten, ohne neue komplette Komponente zu bauen.
  • Wartbarkeit steigt, wenn Selektoren flach und sprechend sind.

Muster / Codeauszug:

.btn { padding:.75rem 1rem; border-radius:.5rem; }
.btn--primary { background:var(--brand); color:#fff; }
.btn--secondary { background:var(--soft); color:var(--brand); }

Prüfung: CSS ist komponentenorientiert und weniger abhängig von starker Verschachtelung.

Schwer: @font-face lokal einbinden und Performance reflektieren

Ziel: Ein lokaler Webfont wird technisch korrekt vorbereitet und mit Fallbacks abgesichert.

  • `font-display:swap` verhindert unsichtbaren Text während des Ladens.
  • Webfonts erhöhen Gestaltungsmöglichkeiten, können aber Ladezeit verursachen.
  • Lizenz und Dateigröße müssen in echten Projekten beachtet werden.

Muster / Codeauszug:

@font-face {
  font-family: "CourseFont";
  src: url("fonts/coursefont.woff2") format("woff2");
  font-display: swap;
}
body { font-family:"CourseFont", Arial, sans-serif; }

Prüfung: @font-face ist syntaktisch korrekt und Fallbacks sind weiterhin vorhanden.

Ziel: Die Teilnehmenden prüfen das gesamte Akkordeon-Projekt auf Struktur, Bedienbarkeit, Responsiveness und CSS-Wartbarkeit.

  • Nachbesserungen sollten nicht nur optisch, sondern technisch begründet sein.

Muster / Codeauszug:

Prüfung: Abschlusscheck ist nachvollziehbar und führt zu konkreten Verbesserungen.

Ziel: Das fertige Akkordeon wird so aufbereitet, dass es im Unterricht als Musterstand gezeigt werden kann.
