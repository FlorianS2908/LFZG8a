Tag 1

Lösungen zu Tag 1: HTML-Grundstruktur und CSS-Basics

Fundament für die Wunderland-Seite: sauberes HTML-Grundgerüst, semantische Bereiche, Textstruktur, Listen, Links, Bilder und CSS-Basisregeln.

Aufgaben Tag 1
Normal: Startseite für Wunderland sauber anlegen

Ziel: Die Teilnehmenden erstellen eine vollständige HTML-Startseite für den Freizeitpark Wunderland.

  • Der technische Kopf gehört in den `head`; sichtbare Inhalte gehören in den `body`.
  • Der Viewport-Meta-Tag ist Grundlage für die späteren Responsive-Aufgaben.
  • Eine eindeutige h1 beschreibt das zentrale Seitenthema.

Muster / Codeauszug:

<!doctype html>
<html lang="de">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Wunderland Freizeitpark</title>
</head>
<body>
  <header class="site-header">
    <p class="logo">Wunderland</p>
    <h1>Willkommen im Wunderland</h1>
  </header>
  <main>
    <p>Entdecke Attraktionen, Preise und magische Momente.</p>
  </main>
</body>
</html>

Prüfung: Die Datei öffnet sich im Browser, der Tab-Titel ist sinnvoll und die HTML-Grundstruktur ist vollständig.

Normal: Seitenbereiche semantisch strukturieren

Ziel: Die Wunderland-Seite wird mit header, main, section, article und footer fachlich sauber gegliedert.

  • `section` ist ein thematischer Abschnitt.
  • `article` passt für einzelne Attraktionskarten, die eigenständig verständlich sind.
  • Der footer bündelt Zusatzinformationen zur Seite.

Muster / Codeauszug:

<main>
  <section class="attractions" aria-labelledby="attraktionen-title">
    <h2 id="attraktionen-title">Unsere Attraktionen</h2>
    <article class="attraction-card">
      <h3>Achterbahn</h3>
      <p>Rasanter Fahrspaß für Mutige.</p>
    </article>
  </section>
</main>

Prüfung: Die Struktur ist nicht nur aus div-Containern gebaut und jeder Bereich hat eine nachvollziehbare Bedeutung.

Normal: Text, Listen und Links für eine Attraktionskarte einbauen

Ziel: Eine Attraktionskarte erhält semantische Textelemente, Stichpunkte und einen internen Link.

  • Eine ul ist besser als Bindestriche im Absatz.
  • Ein Link sollte nicht nur „hier“ heißen.
  • Inhalte sind dadurch besser lesbar und leichter zu gestalten.

Muster / Codeauszug:

<article class="attraction-card" id="achterbahn">
  <h3>Achterbahn</h3>
  <p>Die Achterbahn ist die schnellste Attraktion im Wunderland.</p>
  <ul>
    <li>hohe Geschwindigkeit</li>
    <li>Looping-Bereich</li>
    <li>ab 12 Jahren empfohlen</li>
  </ul>
  <a href="#preise">Zu den Preisen</a>
</article>

Prüfung: Absatz, Liste und Link werden passend eingesetzt; der Linktext sagt klar, wohin er führt.

Normal: Erste CSS-Basisregeln für Wunderland setzen

Ziel: Die Teilnehmenden gestalten Grundschrift, Hintergrund, Abstände und Kartenrahmen.

  • Globale Regeln gehören an `body`, komponentenbezogene Regeln an Klassen.
  • Padding erzeugt Innenabstand, border den Rahmen und margin Außenabstand.
  • Wiederholte Attraktionskarten profitieren von einer gemeinsamen Klasse.

Muster / Codeauszug:

body {
  margin: 0;
  font-family: Arial, sans-serif;
  color: #173248;
  background: #eef8fb;
}
.site-header {
  padding: 2rem;
  background: #003964;
  color: white;
}
.attraction-card {
  padding: 1rem;
  border: 1px solid #d8e8ee;
  border-radius: .75rem;
  background: white;
}

Prüfung: Die Seite wirkt sichtbar gestaltet und die Box-Modell-Eigenschaften sind bewusst eingesetzt.

Normal: Klassen und IDs im Wunderland-Projekt unterscheiden

Ziel: Die Teilnehmenden verwenden Klassen für wiederholbare Gestaltung und IDs für eindeutige Anker.

  • Klassen steuern wiederholbare Komponenten.
  • IDs eignen sich für eindeutige Anker wie `#preise`.
  • Mehrfach verwendete IDs sind fehleranfällig und ungültig.

Muster / Codeauszug:

<section class="attractions" id="attraktionen">
  <article class="attraction-card" id="achterbahn">...</article>
  <article class="attraction-card" id="riesenrad">...</article>
</section>
<a href="#attraktionen">Attraktionen ansehen</a>

Prüfung: Klassen sind mehrfach verwendbar; IDs sind eindeutig und werden nicht doppelt vergeben.

Schwer: Div-lastige Wunderland-Seite semantisch refactoren

Ziel: Eine unsaubere HTML-Struktur wird fachlich sinnvoll in semantische Elemente überführt.

  • Semantik verbessert Wartbarkeit und Barrierefreiheit.
  • Nicht jeder div ist falsch, aber fachliche Bereiche brauchen passende Elemente.
  • CSS-Klassen können auch auf semantischen Elementen verwendet werden.

Muster / Codeauszug:

<!-- vorher: <div class="card"> -->
<article class="attraction-card">
  <h3>Riesenrad</h3>
  <p>Blick über den ganzen Park.</p>
</article>

Prüfung: Die Seite ist semantisch klarer, ohne dass Inhalt verloren geht.

Schwer: Mini-Designsystem mit CSS Custom Properties erstellen

Ziel: Farben, Abstände und Rundungen werden zentral für die Wunderland-Seite verwaltet.

  • Custom Properties reduzieren doppelte Werte.
  • Sprechende Namen wie `--brand` oder `--space` sind verständlicher als Zufallsnamen.
  • Ein Farbsystem macht die Seite konsistenter.

Muster / Codeauszug:

:root {
  --brand: #003964;
  --accent: #00abc7;
  --surface: #ffffff;
  --line: #d8e8ee;
  --radius: .75rem;
  --space: 1rem;
}
.attraction-card {
  background: var(--surface);
  border: 1px solid var(--line);
  border-radius: var(--radius);
  padding: var(--space);
}

Prüfung: Mindestens drei Komponenten nutzen dieselben zentralen Variablen.

Schwer: HTML-Fehleranalyse mit Browser-DevTools durchführen

Ziel: Die Teilnehmenden finden und korrigieren typische Struktur- und Verschachtelungsfehler.

  • Browser reparieren HTML oft still, zeigen aber im DOM die tatsächliche Struktur.
  • Doppelte IDs erschweren Sprungmarken und JavaScript/CSS-Zugriffe.
  • Saubere Verschachtelung ist Grundlage für spätere CSS-Regeln.

Muster / Codeauszug:

<section id="preise">
  <h2>Preise</h2>
  <p>Alle Tickets im Überblick.</p>
</section>

Prüfung: Alle Fehler sind behoben und die Dokumentation benennt Ursache und Wirkung.

Schwer: Attraktionsdaten als wiederholbare Kartenstruktur vorbereiten

Ziel: Die HTML-Struktur wird so aufgebaut, dass später mehrere Attraktionen konsistent ergänzt werden können.

  • Wiederholbare Komponenten brauchen gemeinsame Klassennamen.
  • Inhalte unterscheiden sich, Struktur und Gestaltung bleiben gleich.
  • Das ist eine Grundlage für wartbare Frontend-Komponenten.

Muster / Codeauszug:

<article class="attraction-card">
  <h3>Geisterhaus</h3>
  <p>Spannung im Dunkeln.</p>
  <ul><li>Indoor</li><li>ab 10 Jahren</li><li>ca. 8 Minuten</li></ul>
  <a href="html/attraktionen/geisterhaus.html">Details</a>
</article>

Prüfung: Eine sechste Karte kann hinzugefügt werden, ohne eine neue CSS-Klasse zu benötigen.

Schwer: Box-Modell gezielt an Karten, Header und Main erklären

Ziel: Die Teilnehmenden markieren und erklären margin, border, padding und content an mehreren Elementen.

  • `box-sizing:border-box` macht Breiten leichter berechenbar.
  • Padding gehört zur Innenfläche, margin trennt Elemente voneinander.
  • Rahmen können Gestaltung und Debugging unterstützen.

Muster / Codeauszug:

* { box-sizing: border-box; }
main { max-width: 1100px; margin: 2rem auto; padding: 1rem; }
.attraction-card { margin: 0; padding: 1rem; border: 2px solid #00abc7; }

Prüfung: Die Erklärung ist fachlich korrekt und die CSS-Wirkung im Browser sichtbar.

Tag 2

Lösungen zu Tag 2: Navigation, Buttons und Flexbox

Navigation, Buttons, Linklisten und flexible Ausrichtung: Wunderland bekommt eine bedienbare Oberfläche mit Flexbox, Hover- und Fokuszuständen.

Aufgaben Tag 2
Normal: Hauptnavigation für Wunderland bauen

Ziel: Die Teilnehmenden erstellen eine semantische Navigation mit mehreren Bereichen der Freizeitpark-Seite.

  • `nav` kennzeichnet zentrale Navigation.
  • Eine Liste passt gut für mehrere Navigationspunkte.
  • `aria-label` hilft, wenn mehrere Navigationen existieren.

Muster / Codeauszug:

<nav class="main-nav" aria-label="Hauptnavigation">
  <ul>
    <li><a href="#attraktionen">Attraktionen</a></li>
    <li><a href="#preise">Preise</a></li>
    <li><a href="#kontakt">Kontakt</a></li>
  </ul>
</nav>

Prüfung: Die Navigation ist semantisch korrekt und führt zu vorhandenen oder vorbereiteten Ankern.

Normal: Navigation mit Flexbox horizontal ausrichten

Ziel: Die Linkliste wird mit Flexbox ausgerichtet und erhält gleichmäßige Abstände.

  • Flexbox eignet sich gut für eindimensionale Ausrichtung.
  • `gap` ist sauberer als viele Einzel-margins.
  • Listenpunkte können über `list-style:none` entfernt werden.

Muster / Codeauszug:

.main-nav ul {
  display: flex;
  gap: 1rem;
  list-style: none;
  padding: 0;
  margin: 0;
}
.main-nav a {
  text-decoration: none;
  font-weight: 700;
}

Prüfung: Die Navigation steht horizontal, hat saubere Abstände und keine Standard-Bullets.

Normal: Call-to-Action-Button für Tickets gestalten

Ziel: Ein Ticket-Link wird optisch wie ein Button gestaltet, bleibt aber ein Link.

  • Navigation zu einem Ziel bleibt ein `a`-Element.
  • Eine Aktion ohne Ziel wäre eher ein `button`.
  • Hover-Zustände geben visuelles Feedback.

Muster / Codeauszug:

<a class="btn" href="#preise">Tickets ansehen</a>

.btn {
  display: inline-block;
  padding: .8rem 1.1rem;
  background: #00abc7;
  color: white;
  border-radius: 999px;
  text-decoration: none;
}
.btn:hover { background: #003964; }

Prüfung: Der Link sieht wie ein Button aus, navigiert aber semantisch korrekt zu Preisen.

Normal: Header-Layout mit Logo und Navigation ausrichten

Ziel: Logo und Navigation werden im Header mit Flexbox sinnvoll verteilt.

  • Flexbox kann Elemente entlang einer Achse verteilen.
  • `align-items:center` richtet vertikal aus.
  • `flex-wrap` verhindert zu starkes Quetschen.

Muster / Codeauszug:

.header-inner {
  display: flex;
  align-items: center;
  justify-content: space-between;
  flex-wrap: wrap;
  gap: 1rem;
}

Prüfung: Logo und Navigation stehen auf breiten Bildschirmen nebeneinander und brechen bei wenig Platz um.

Normal: Attraktionskarten als flexible Reihe darstellen

Ziel: Mehrere Attraktionskarten werden mit Flexbox nebeneinander und umbrechend dargestellt.

  • Flexbox ist für Reihen von Karten gut geeignet.
  • `flex: 1 1 220px` kombiniert Wachstum, Schrumpfen und Basisbreite.
  • Gleiche Klassen erzeugen konsistente Komponenten.

Muster / Codeauszug:

.card-row {
  display: flex;
  flex-wrap: wrap;
  gap: 1rem;
}
.attraction-card {
  flex: 1 1 220px;
}

Prüfung: Die Karten stehen nebeneinander, brechen aber bei kleiner Breite sauber um.

Schwer: Responsive Navigationslogik ohne JavaScript vorbereiten

Ziel: Die Navigation wird so gebaut, dass sie ohne JavaScript auf kleineren Breiten sauber nutzbar bleibt.

  • Ohne JavaScript ist ein gutes Umbruchverhalten besonders wichtig.
  • Fokuszustände sind für Tastaturbedienung Pflicht.
  • Touchflächen brauchen ausreichend padding.

Muster / Codeauszug:

.main-nav ul { display:flex; flex-wrap:wrap; gap:.5rem; }
.main-nav a { padding:.65rem .85rem; border-radius:999px; }
.main-nav a:focus-visible { outline:3px solid #bdf77a; outline-offset:3px; }

Prüfung: Alle Links bleiben erreichbar, fokussierbar und lesbar.

Schwer: Button-vs-Link-Entscheidung im Wunderland dokumentieren

Ziel: Die Teilnehmenden prüfen mehrere Bedienelemente und entscheiden fachlich zwischen a und button.

  • Ein Link navigiert zu einem Ziel.
  • Ein Button löst eine Aktion aus, z. B. Formular senden oder UI ändern.
  • Optik darf Semantik nicht ersetzen.

Muster / Codeauszug:

<!-- Navigation: Link -->
<a class="btn" href="#preise">Tickets ansehen</a>

<!-- Aktion: Button -->
<button class="btn" type="submit">Anfrage senden</button>

Prüfung: Die Begründungen unterscheiden Navigation und Aktion korrekt.

Schwer: Flexbox-Fehlerdiagnose an einer Kartenreihe durchführen

Ziel: Eine fehlerhafte Kartenreihe wird analysiert und stabil gemacht.

  • Feste Breiten ohne Umbruch führen schnell zu horizontalem Scrollen.
  • `flex-wrap:wrap` und flexible Basisbreiten lösen viele Kartenreihen-Probleme.
  • `min-width` sollte bewusst gesetzt werden.

Muster / Codeauszug:

.card-row { display:flex; flex-wrap:wrap; gap:1rem; }
.attraction-card { flex:1 1 min(260px, 100%); min-width:0; }

Prüfung: Die Karten überlaufen nicht mehr und die Erklärung nennt die wirksamen Flex-Eigenschaften.

Schwer: Komponentenklassen für Navigation und Karten refactoren

Ziel: CSS wird von Tag-Selektoren auf wartbare Komponentenklassen umgestellt.

  • Tag-Selektoren sind für globale Grundlagen sinnvoll, aber für Komponenten oft zu breit.
  • Komponentenklassen machen CSS besser wartbar.
  • BEM-ähnliche Namen können Zusammenhänge sichtbar machen.

Muster / Codeauszug:

.main-nav__link { padding:.65rem .85rem; border-radius:999px; }
.attraction-card { padding:1rem; border:1px solid #d8e8ee; }
.attraction-card__title { margin:0 0 .35rem; }

Prüfung: Die Gestaltung trifft gezielt die Wunderland-Komponenten und nicht versehentlich alle gleichnamigen Tags.

Schwer: Header mit mehreren Flex-Zonen bauen

Ziel: Der Wunderland-Header bekommt Logo, Navigation und Ticket-CTA in einer robusten Flex-Struktur.

  • Mehrere Flex-Zonen brauchen klare Wachstums- und Umbruchregeln.
  • Ein CTA darf die Navigation nicht verdrängen.
  • `gap` und `flex-wrap` verbessern Stabilität.

Muster / Codeauszug:

.header-inner { display:flex; align-items:center; gap:1rem; flex-wrap:wrap; }
.main-nav { margin-left:auto; }
.ticket-cta { flex:0 0 auto; }
@media (max-width:700px){ .main-nav { margin-left:0; width:100%; } }

Prüfung: Header bleibt lesbar, Navigation und CTA kollidieren nicht und brechen sauber um.

Tag 3

Lösungen zu Tag 3: Struktur, Karten und Grid

Mehrspaltige Struktur für Wunderland: Kartenraster, Galerie, Preisübersicht, aside-Bereiche und Grid-Layouts.

Aufgaben Tag 3
Normal: Attraktionsraster mit CSS Grid erstellen

Ziel: Attraktionskarten werden in einem einfachen Grid angeordnet.

  • Grid eignet sich für zweidimensionale Raster.
  • `auto-fit` und `minmax` reduzieren Medienabfragen.
  • Karten behalten durch gemeinsame Klasse ein einheitliches Layout.

Muster / Codeauszug:

.attraction-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  gap: 1rem;
}

Prüfung: Das Raster passt sich der Breite an und zeigt Karten in mehreren Spalten.

Normal: Galerie mit figure und figcaption aufbauen

Ziel: Bilder und Beschriftungen werden semantisch zusammengehalten.

  • `figure` gruppiert Medieninhalt und Beschriftung.
  • `figcaption` beschreibt das Bild kurz.
  • Bilder sollten responsive bleiben.

Muster / Codeauszug:

<figure class="gallery-card">
  <img src="images/galerie/riesenrad.jpg" alt="Riesenrad im Wunderland">
  <figcaption>Riesenrad bei Sonnenuntergang</figcaption>
</figure>

.gallery-card img { max-width:100%; height:auto; display:block; }

Prüfung: Jedes Bild besitzt eine passende Beschriftung und ist im Grid angeordnet.

Normal: Aside-Infobox für Öffnungszeiten ergänzen

Ziel: Eine ergänzende Infobox wird neben dem Hauptinhalt platziert.

  • `aside` passt für Zusatzinformationen.
  • Optisch sollte die Box abgesetzt, aber nicht dominanter als der Hauptinhalt sein.
  • Auf kleinen Bildschirmen darf der aside unter dem Inhalt stehen.

Muster / Codeauszug:

<aside class="info-box">
  <h2>Öffnungszeiten</h2>
  <p>Täglich von 9:00 bis 18:00 Uhr.</p>
</aside>

Prüfung: Der aside ist eindeutig ergänzend und nicht der zentrale Hauptinhalt.

Normal: Preis-Karten in einem Grid darstellen

Ziel: Ticketpreise werden als übersichtliche Karten dargestellt.

  • Preiskarten sind eigenständige Inhalte und passen zu `article`.
  • Ein hervorgehobener Tarif kann eine zusätzliche Klasse bekommen.
  • Grid verhindert ungleichmäßige Reihen bei breiter Ansicht.

Muster / Codeauszug:

<section id="preise" class="price-grid">
  <article class="price-card price-card--featured">
    <h3>Familienticket</h3>
    <p>4 Personen · 79 €</p>
  </article>
</section>

Prüfung: Die Preise sind als Karten erkennbar, gleichmäßig angeordnet und gut lesbar.

Normal: Footer in Spalten gliedern

Ziel: Der Wunderland-Footer erhält mehrere Informationsspalten.

  • Grid ist für Footer-Spalten gut geeignet.
  • `repeat(auto-fit, minmax())` macht das Layout flexibel.
  • Footer-Links sollten lesbar und klickbar bleiben.

Muster / Codeauszug:

.site-footer__grid {
  display:grid;
  grid-template-columns: repeat(auto-fit, minmax(180px, 1fr));
  gap: 1rem;
}

Prüfung: Footer-Inhalte stehen bei breiter Ansicht nebeneinander und bei schmaler Ansicht untereinander.

Schwer: Komplettes Seitenlayout mit Grid Areas bauen

Ziel: Header, Navigation, main, aside und footer werden über Grid Areas angeordnet.

  • Grid Areas machen Seitenlayouts sehr lesbar.
  • Die mobile Struktur sollte linear und logisch bleiben.
  • Benannte Areas sind für Unterricht gut erklärbar.

Muster / Codeauszug:

.page-layout {
  display:grid;
  grid-template-columns: 1fr 280px;
  grid-template-areas:
    "header header"
    "nav nav"
    "main aside"
    "footer footer";
  gap:1rem;
}
main{grid-area:main}.info-box{grid-area:aside}
@media(max-width:800px){ .page-layout{grid-template-columns:1fr; grid-template-areas:"header" "nav" "main" "aside" "footer";} }

Prüfung: Das Layout nutzt benannte Bereiche und bleibt auf kleinen Breiten nutzbar.

Schwer: Galerie mit unterschiedlich großen Kacheln erstellen

Ziel: Die Galerie erhält einzelne hervorgehobene Kacheln über Grid-Spalten und -Zeilen.

  • Hervorgehobene Grid-Kacheln erzeugen visuelle Hierarchie.
  • Bei kleiner Breite sollten Spezialspannen zurückgesetzt werden.
  • Inhalt darf nicht von Gestaltung abhängig sein.

Muster / Codeauszug:

.gallery-grid{display:grid;grid-template-columns:repeat(auto-fit,minmax(180px,1fr));gap:1rem}
.gallery-card--wide{grid-column:span 2}
@media(max-width:600px){.gallery-card--wide{grid-column:auto}}

Prüfung: Die Galerie wirkt dynamisch, bleibt aber responsive und lesbar.

Schwer: Grid-vs-Flexbox im Wunderland begründet vergleichen

Ziel: Die Teilnehmenden entscheiden, welche Layouttechnik für welche Stelle passender ist.

  • Flexbox eignet sich für Reihen oder Spalten.
  • Grid eignet sich für Raster und Seitenbereiche.
  • Gute Frontends kombinieren beide Techniken.

Muster / Codeauszug:

/* Navigation: Flexbox */
.main-nav ul{display:flex;gap:1rem}
/* Kartenübersicht: Grid */
.attraction-grid{display:grid;grid-template-columns:repeat(auto-fit,minmax(220px,1fr));gap:1rem}

Prüfung: Die Begründung nennt Achsenlogik: eindimensional Flexbox, zweidimensional Grid.

Schwer: Preistabelle in Kartenlayout refactoren

Ziel: Eine starre Tabelle wird für eine Marketing-Seite in Karten umgewandelt, ohne Informationen zu verlieren.

  • Tabellen sind gut für Datenvergleich, Karten gut für präsentative Angebote.
  • Struktur darf beim Refactoring keine Inhalte verlieren.
  • Jede Karte braucht klare Überschrift und Preisangabe.

Muster / Codeauszug:

<article class="price-card">
  <h3>Kinderticket</h3>
  <p class="price">24 €</p>
  <p>Gültig bis 14 Jahre.</p>
</article>

Prüfung: Alle Informationen bleiben erhalten und das Kartenlayout ist für die Zielseite verständlich.

Schwer: Grid-Layout debuggen: ungleichmäßige Kartenhöhen lösen

Ziel: Ungleich lange Texte verursachen optische Probleme im Raster und werden sauber behandelt.

  • Feste Höhen sind oft fragile Lösungen.
  • Eine Karte kann intern `display:flex` und `margin-top:auto` nutzen.
  • So bleibt die Karte flexibel bei Textlängen.

Muster / Codeauszug:

.attraction-card{display:flex;flex-direction:column}
.attraction-card .btn{margin-top:auto;align-self:flex-start}

Prüfung: Buttons stehen konsistent am Kartenende, ohne feste Höhen zu erzwingen.

Tag 4

Lösungen zu Tag 4: Responsive HTML-Elemente

Responsive Wunderland-Seite: Mobile First, Breakpoints, flexible Bilder, responsive Navigation und robuste Breiten mit min(), max(), clamp() und media queries.

Aufgaben Tag 4
Normal: Viewport und mobile Grundprüfung durchführen

Ziel: Die Teilnehmenden prüfen, ob die Wunderland-Seite korrekt für mobile Ansichten vorbereitet ist.

  • Ohne Viewport-Meta-Tag funktioniert Mobile First nicht zuverlässig.
  • DevTools helfen beim Testen verschiedener Breiten.
  • Probleme sollten konkret benannt werden: Überlauf, zu kleine Touchflächen oder feste Breiten.

Muster / Codeauszug:

<meta name="viewport" content="width=device-width, initial-scale=1">

Prüfung: Die Seite ist in mobiler Ansicht nicht künstlich verkleinert und mindestens ein Layoutproblem ist behoben.

Normal: Bilder in der Galerie flexibel machen

Ziel: Galeriebilder werden responsive dargestellt und verursachen keinen horizontalen Scrollbalken.

  • `max-width:100%` verhindert Überlauf.
  • `height:auto` erhält das Seitenverhältnis.
  • `display:block` verhindert kleine Inline-Abstände unter Bildern.

Muster / Codeauszug:

img {
  max-width: 100%;
  height: auto;
  display: block;
}

Prüfung: Bilder passen sich dem Container an und behalten ihr Seitenverhältnis.

Normal: Navigation bei kleiner Breite umbrechen lassen

Ziel: Die Navigation bleibt auch auf kleinen Bildschirmen bedienbar.

  • Eine horizontale Desktopnavigation darf mobil umbrechen.
  • Klickflächen brauchen ausreichend Innenabstand.
  • Umbruch ist ohne JavaScript eine solide Basislösung.

Muster / Codeauszug:

.main-nav ul{display:flex;flex-wrap:wrap;gap:.5rem}
.main-nav a{display:block;padding:.7rem .9rem}

Prüfung: Alle Navigationslinks bleiben lesbar und klickbar.

Normal: Schriftgrößen mit clamp() flexibel setzen

Ziel: Überschriften skalieren zwischen mobilen und großen Ansichten.

  • `clamp(min, bevorzugt, max)` begrenzt flexible Werte.
  • So werden weniger Breakpoints benötigt.
  • Lesbarkeit steht vor reiner Optik.

Muster / Codeauszug:

h1 {
  font-size: clamp(2rem, 6vw, 4rem);
  line-height: 1.05;
}

Prüfung: Die Überschrift ist auf kleinen Geräten nicht zu groß und auf großen nicht zu klein.

Normal: Responsive Kartenbreiten mit min() oder minmax() setzen

Ziel: Karten erhalten flexible Breiten, ohne aus dem Layout zu laufen.

  • Flexible CSS-Funktionen reduzieren Sonderfälle.
  • `min(100%, 320px)` verhindert Überlauf.
  • Grid mit minmax ist für Kartenraster oft noch besser.

Muster / Codeauszug:

.info-box { width: min(100%, 320px); }
.attraction-grid { grid-template-columns: repeat(auto-fit, minmax(220px, 1fr)); }

Prüfung: Karten überlaufen nicht und wirken auf breiten Ansichten trotzdem kontrolliert.

Schwer: Mobile-First-Refactoring der Wunderland-Seite durchführen

Ziel: Ein Desktop-lastiges CSS wird in Mobile-First-CSS umgebaut.

  • Mobile First startet mit der einfachsten linearen Ansicht.
  • Media Queries erweitern Layout, statt es zu reparieren.
  • Feste Breiten sind häufig Ursache für mobile Fehler.

Muster / Codeauszug:

.page { padding: 1rem; }
.attraction-grid { display:grid; gap:1rem; }
@media (min-width: 760px) {
  .attraction-grid { grid-template-columns: repeat(3, 1fr); }
}

Prüfung: Die mobile Ansicht funktioniert ohne große Media Query und Desktop wird schrittweise erweitert.

Schwer: Responsive picture-Element für Hero-Bild einbauen

Ziel: Das Hero-Bild nutzt je nach Breite unterschiedliche Quellen.

  • `picture` bietet alternative Bildquellen.
  • `source` selbst ist nicht sichtbar.
  • Das `img` bleibt wichtig für Fallback und alt-Text.

Muster / Codeauszug:

<picture>
  <source media="(min-width: 900px)" srcset="images/hero-wide.jpg">
  <source media="(min-width: 600px)" srcset="images/hero-tablet.jpg">
  <img src="images/hero-mobile.jpg" alt="Eingang des Wunderland-Freizeitparks">
</picture>

Prüfung: Das picture-Element ist korrekt aufgebaut und das fallback-img ist vorhanden.

Schwer: Breakpoints fachlich begründen und reduzieren

Ziel: Die Teilnehmenden prüfen vorhandene Breakpoints und entfernen unnötige.

  • Nicht jede Bildschirmbreite braucht eigene Regeln.
  • Flexible Layouttechniken reduzieren Breakpoint-Komplexität.
  • Breakpoints sollten am Layoutproblem entstehen, nicht an Gerätenamen.

Muster / Codeauszug:

.cards {
  display:grid;
  grid-template-columns: repeat(auto-fit, minmax(240px, 1fr));
  gap:1rem;
}
/* ersetzt mehrere starre 768/1024-Regeln */

Prüfung: Mindestens eine Media Query wurde sinnvoll entfernt oder ersetzt.

Schwer: Responsive Abnahmecheck für Wunderland erstellen

Ziel: Die Teilnehmenden erstellen einen fachlichen Testplan für responsive Darstellung.

  • Responsive Qualität muss getestet werden, nicht nur „gefühlt“ passen.
  • Konkrete Breiten machen Fehler reproduzierbar.
  • Prüfpunkte sollten sichtbare und bedienbare Aspekte abdecken.

Muster / Codeauszug:

<table>
  <tr><th>Breite</th><th>Prüfpunkt</th><th>Ergebnis</th></tr>
  <tr><td>360px</td><td>Navigation klickbar</td><td>OK</td></tr>
</table>

Prüfung: Der Testplan enthält klare Prüfpunkte und nachvollziehbare Ergebnisse.

Schwer: Horizontales Scrollen systematisch finden und beheben

Ziel: Eine Seite mit verstecktem Überlauf wird analysiert.

  • `overflow:hidden` versteckt oft nur das Symptom.
  • Bilder, feste Breiten und lange Inhalte sind typische Ursachen.
  • `min-width:0` kann in Grid/Flex-Kontexten wichtig sein.

Muster / Codeauszug:

.card { min-width: 0; }
.card img { max-width: 100%; height: auto; }
.long-text { overflow-wrap: anywhere; }

Prüfung: Der Scrollbalken ist weg und die Ursache wurde korrekt behoben.

Tag 5

Lösungen zu Tag 5: Formulare, Webfonts und CSS-Struktur

Formulare, Fokuszustände, Webfonts und CSS-Struktur: Wunderland bekommt Kontakt-/Buchungsformular, klare Labels, sichtbare Bedienzustände und wartbares CSS.

Aufgaben Tag 5
Normal: Kontaktformular für Wunderland erstellen

Ziel: Die Teilnehmenden bauen ein semantisches Kontaktformular mit Labels und passenden Eingabetypen.

  • Labels verbessern Bedienbarkeit und Barrierefreiheit.
  • `type="email"` unterstützt Validierung und mobile Tastaturen.
  • Ein Submit-Button braucht einen klaren Text.

Muster / Codeauszug:

<form class="contact-form">
  <label for="email">E-Mail</label>
  <input id="email" name="email" type="email" required>
  <label for="message">Nachricht</label>
  <textarea id="message" name="message" rows="5"></textarea>
  <button type="submit">Anfrage senden</button>
</form>

Prüfung: Alle Felder haben Labels und passende type-Werte.

Normal: Formularfelder sichtbar und einheitlich gestalten

Ziel: Inputs, Textarea und Select erhalten ein konsistentes Styling.

  • Formularfelder sollten zur Schrift der Seite passen.
  • Konsistente Abstände machen Formulare ruhiger.
  • Volle Breite im Formular ist mobil oft am einfachsten.

Muster / Codeauszug:

.contact-form input,
.contact-form textarea,
.contact-form select {
  width: 100%;
  font: inherit;
  padding: .75rem;
  border: 2px solid #d8e8ee;
  border-radius: .5rem;
}

Prüfung: Formularfelder wirken einheitlich und sind gut bedienbar.

Normal: Sichtbare Fokuszustände ergänzen

Ziel: Formularfelder und Buttons erhalten gut erkennbare Tastatur-Fokuszustände.

  • Fokuszustände sind essenziell für Bedienbarkeit.
  • Outline sollte nicht komplett entfernt werden.
  • `:focus-visible` unterscheidet oft sinnvoll zwischen Maus- und Tastaturfokus.

Muster / Codeauszug:

a:focus-visible, button:focus-visible, input:focus-visible, textarea:focus-visible {
  outline: 3px solid #00abc7;
  outline-offset: 3px;
}

Prüfung: Tastaturbedienung zeigt jederzeit klar, welches Element aktiv ist.

Normal: Auswahlfeld für Besuchsanlass bauen

Ziel: Ein Select-Feld wird sinnvoll beschriftet und mit Optionen gefüllt.

  • Select eignet sich bei mehreren Auswahloptionen.
  • Der sichtbare Text kann nutzerfreundlich sein; value darf technisch sein.
  • Bei wenigen Optionen könnten Radio-Buttons eine Alternative sein.

Muster / Codeauszug:

<label for="visit-reason">Anlass des Besuchs</label>
<select id="visit-reason" name="visitReason">
  <option value="familie">Familienausflug</option>
  <option value="schule">Schulklasse</option>
  <option value="geburtstag">Geburtstag</option>
</select>

Prüfung: Select und Label sind korrekt verbunden und verständlich.

Normal: CSS-Datei logisch strukturieren

Ziel: Das Wunderland-CSS wird in nachvollziehbare Abschnitte gegliedert.

  • Struktur ist ein Qualitätsmerkmal.
  • Kommentare sollen Abschnitte erklären, nicht jede einzelne Zeile.
  • Doppelte Regeln erschweren Wartung.

Muster / Codeauszug:

/* 1. Grundregeln */
body { ... }

/* 2. Layout */
.site-header { ... }

/* 3. Komponenten */
.attraction-card { ... }

/* 4. Formular */
.contact-form { ... }

Prüfung: CSS ist lesbarer und enthält keine offensichtlichen Duplikate.

Schwer: Buchungsformular mit fieldset und legend strukturieren

Ziel: Ein umfangreicheres Formular wird in sinnvolle Gruppen gegliedert.

  • `fieldset` gruppiert zusammengehörige Felder.
  • `legend` benennt die Gruppe.
  • Passende input-Typen verbessern Bedienbarkeit und Validierung.

Muster / Codeauszug:

<fieldset>
  <legend>Besuchsdaten</legend>
  <label for="date">Besuchsdatum</label>
  <input id="date" name="date" type="date">
  <label for="people">Personenanzahl</label>
  <input id="people" name="people" type="number" min="1">
</fieldset>

Prüfung: Das Formular ist semantisch gruppiert und die Gruppen sind visuell erkennbar.

Schwer: Fehler- und Pflichtfeldzustände gestalten

Ziel: Formularzustände werden sichtbar, aber nicht nur über Farbe kommuniziert.

  • Farbe allein reicht nicht als Fehlerkommunikation.
  • Hinweistext und Rahmen helfen zusätzlich.
  • `aria-invalid` kann Assistenztechnologien unterstützen.

Muster / Codeauszug:

<label for="mail">E-Mail <span aria-hidden="true">*</span></label>
<input id="mail" type="email" aria-invalid="true">
<p class="field-error">Bitte eine gültige E-Mail-Adresse eingeben.</p>

[aria-invalid="true"] { border-color:#b42318; }
.field-error { font-weight:700; }

Prüfung: Fehlerzustände sind sichtbar, verständlich und barriereärmer umgesetzt.

Schwer: Webfont-Konzept mit Fallbacks planen

Ziel: Die Teilnehmenden integrieren oder simulieren ein Font-Konzept, ohne die Seite von einer einzigen Schrift abhängig zu machen.

  • Fallbacks sind wichtig, wenn Webfonts nicht geladen werden.
  • Schmuckschriften gehören sparsam eingesetzt.
  • Für Fließtext zählt Lesbarkeit mehr als Effekt.

Muster / Codeauszug:

:root {
  --font-heading: "Lobster Two", Georgia, serif;
  --font-body: "Open Sans", Arial, sans-serif;
}
body { font-family: var(--font-body); }
h1, h2 { font-family: var(--font-heading); }

Prüfung: Schriften sind mit Fallbacks definiert und die Seite bleibt lesbar.

Schwer: CSS-Komponenten und Utilities trennen

Ziel: Das CSS wird so refactort, dass Layout-Hilfen und Komponentenregeln nicht vermischt werden.

  • Komponenten beschreiben konkrete UI-Teile.
  • Utilities sind kleine wiederverwendbare Helfer.
  • Zu starke Utilities können Wartungsprobleme erzeugen.

Muster / Codeauszug:

.container { width:min(1100px, 100% - 2rem); margin-inline:auto; }
.flow > * + * { margin-top:1rem; }
.btn { display:inline-flex; align-items:center; padding:.75rem 1rem; }
.attraction-card { padding:1rem; border:1px solid var(--line); }

Prüfung: Komponenten und Hilfsklassen sind logisch getrennt und überschreiben sich nicht unkontrolliert.

Ziel: Die fertigen Zusatzaufgaben werden als prüfbarer Projektstand zusammengeführt.

  • Eine Abnahme bündelt technische und fachliche Prüfpunkte.
