Tag 1 · CSS-Basis · Custom Properties · Lösungen

CSS-Grundlagen, Einbindung und erste Komponenten

Bestehende Tageslösung Tag 1 Teilnehmeraufgaben Tag 1

01 Grundgerüst, Ordnerstruktur und CSS-Einbindung · Musterlösung: Grundgerüst und CSS-Datei

Die Lösung trennt HTML-Struktur und Gestaltung. Im HTML steht nur der Link zur externen CSS-Datei. Der Pfad ist relativ zur HTML-Datei.

<!doctype html>
<html lang="de">
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>Profilkarte</title>
  <link rel="stylesheet" href="css/style.css">
</head>
<body>
  <main class="page">
    <section class="profile-card">...</section>
  </main>
</body>
</html>

Abnahmekriterium: Die Seite startet fehlerfrei, der Link zur CSS-Datei ist relativ und Styling steht ausschließlich in der externen CSS-Datei.

02 Reset, Custom Properties und Designwerte · Musterlösung: Reset und Variablen

box-sizing erleichtert die Größenberechnung. Custom Properties bündeln wiederkehrende Designwerte und reduzieren doppelte Werte.

:root {
  --color-primary: #003964;
  --color-accent: #00abc7;
  --color-text: #163044;
  --color-bg: #f3fbff;
  --space-sm: .5rem;
  --space-md: 1rem;
  --space-lg: 2rem;
}
* { box-sizing: border-box; }

Abnahmekriterium: Farben und Abstände lassen sich zentral im :root ändern; doppelte Hexwerte sind weitgehend entfernt.

03 Selektoren, Kaskade und Spezifität sichtbar machen · Musterlösung: Spezifität erklären

Bei gleicher Eigenschaft gewinnt die spezifischere Regel. Eine ID ist stärker als eine Klasse, eine Klasse stärker als ein Elementselektor. !important wird nicht als normale Lösung genutzt.

h1 { color: var(--color-text); }
.card { color: var(--color-primary); }
#main-title { color: var(--color-accent); }
.card > p { line-height: 1.6; }

Abnahmekriterium: Die Teilnehmenden können erklären, warum ID vor Klasse und Klasse vor Typselektor wirkt.

04 Profilkarte als erste wiederverwendbare Komponente · Musterlösung: Profilkarte

Die Profilkarte wird als Komponente gebaut. Sie nutzt max-width statt starrer Breiten und arbeitet mit Variablen für Farben und Abstände.

.profile-card {
  max-width: 28rem;
  padding: var(--space-lg);
  border: 1px solid #d8e8ee;
  border-radius: 1rem;
  background: #fff;
}
.button {
  display: inline-block;
  padding: .75rem 1rem;
  border-radius: .5rem;
  background: var(--color-primary);
  color: #fff;
}
.button:hover { background: #005180; }

Abnahmekriterium: Die Card bleibt lesbar, nutzt nachvollziehbare Klassennamen und läuft bei schmaler Breite nicht aus dem Viewport.

05 Fehleranalyse, Lesbarkeit und Zwischenabgabe · Musterlösung: Fehleranalyse und Abgabe

Typische Fehler sind falsche Pfade, vergessene Semikolons, fehlende geschweifte Klammern oder falsch geschriebene Variablen. Die Abgabe muss im Browser ohne sichtbare Strukturfehler starten.

/* Strukturvorschlag */
/* 1. Basis */
/* 2. Layout */
/* 3. Komponenten */
/* 4. Hilfsklassen */

Abnahmekriterium: Der Zwischenstand ist lauffähig, fachlich erklärbar und in der CSS-Datei sauber gegliedert.

Lösungshinweise zu den Zusatzaufgaben Tag 1

Die vorhandenen Zusatzaufgaben sind nach Tag und Thema einsortiert. Wenn eine konkrete Lösungsdatei vorhanden ist, ist sie in der Aufgabenübersicht direkt verlinkt. Für Aufgaben ohne eindeutige Datei gelten diese fachlichen Lösungskriterien:

  • HTML-Grundgerüst vollständig: doctype, html lang, head, meta charset, viewport, title und body.
  • CSS extern eingebunden; keine absoluten Pfade.
  • Box-Modell, Farben, Abstände, Selektoren und Vererbung müssen am Ergebnis erklärbar sein.
Tag 2 · Flexbox · Navigation · Karten · Lösungen

Flexbox, Navigation und Komponenten

Bestehende Tageslösung Tag 2 Teilnehmeraufgaben Tag 2

01 Navigation mit Flexbox strukturieren · Musterlösung: Flex-Navigation

Der gemeinsame Container wird zum Flex-Container. Links werden nicht einzeln verschoben, sondern über gap und Ausrichtung gesteuert.

.site-nav {
  display: flex;
  align-items: center;
  justify-content: space-between;
  gap: 1rem;
}
.nav-links {
  display: flex;
  gap: .75rem;
  flex-wrap: wrap;
}

Abnahmekriterium: Die Navigation ist horizontal ausgerichtet, sauber verteilt und ohne viele einzelne Margin-Sonderregeln gebaut.

02 Kurskarten als wiederverwendbare Flex-Komponenten · Musterlösung: Flex-Card

Die Card selbst kann eine Spalte sein. Mit margin-top:auto oder flexibler Struktur bleibt der Aktionsbereich unten.

.course-card {
  display: flex;
  flex-direction: column;
  gap: .75rem;
}
.course-card__actions {
  margin-top: auto;
}

Abnahmekriterium: Die Karten wirken einheitlich, unterschiedliche Textlängen zerstören das Layout nicht und Komponentenklassen sind wiederverwendbar.

03 Flexible Breiten, Umbruch und Achsen erklären · Musterlösung: flexible Kartenreihe

flex-wrap erlaubt Umbruch. flex: 1 1 260px gibt eine Zielbreite, erlaubt aber Verkleinern und Wachsen.

.course-grid {
  display: flex;
  flex-wrap: wrap;
  gap: 1rem;
}
.course-card {
  flex: 1 1 260px;
}

Abnahmekriterium: Bei kleiner Breite brechen Karten kontrolliert um; die Achsen können am eigenen Code erklärt werden.

04 Interaktion, Hover, Fokus und Pseudo-Klassen · Musterlösung: Hover und Fokus

Hover ist nur Mausfeedback. Für Tastaturbedienung braucht es zusätzlich einen sichtbaren Fokuszustand.

.nav-links a:hover {
  background: #e6fbff;
}
.nav-links a:focus-visible,
.button:focus-visible {
  outline: 3px solid #00abc7;
  outline-offset: 3px;
}

Abnahmekriterium: Links und Buttons geben visuelles Feedback, bleiben bedienbar und erfüllen einfache Zugänglichkeitsanforderungen.

05 Responsive Navigation, Screenshots und Peer Review · Musterlösung: responsive Navigation

Auf kleinen Breiten darf die Navigation gestapelt werden. Der Breakpoint wird gesetzt, wenn die horizontale Navigation zu eng wird.

@media (max-width: 520px) {
  .site-nav, .nav-links {
    flex-direction: column;
    align-items: stretch;
  }
}

Abnahmekriterium: Die Seite funktioniert auch mobil, Screenshots liegen vor und der Einsatz von Flexbox wurde kurz begründet.

Lösungshinweise zu den Zusatzaufgaben Tag 2

Die vorhandenen Zusatzaufgaben sind nach Tag und Thema einsortiert. Wenn eine konkrete Lösungsdatei vorhanden ist, ist sie in der Aufgabenübersicht direkt verlinkt. Für Aufgaben ohne eindeutige Datei gelten diese fachlichen Lösungskriterien:

  • Navigation und Kartenlayout mit Flexbox umsetzen, nicht mit willkürlichen Einzelabständen.
  • Pseudo-Klassen nur dort einsetzen, wo sie Bedienzustände oder Struktur sinnvoll zeigen.
  • Dropdown- oder komplexere Navigation muss auch ohne Layoutbruch lesbar bleiben.
Tag 3 · Grid · Dashboard · Tabellen · Lösungen

CSS Grid, Dashboard und Datenlayout

Bestehende Tageslösung Tag 3 Teilnehmeraufgaben Tag 3

01 Dashboard-Grundstruktur mit semantischen Bereichen · Musterlösung: Dashboard-Struktur

Die Bereiche werden semantisch aufgebaut. Das erleichtert Lesbarkeit, Wartung und spätere Layoutzuweisung per Grid Area.

<div class="dashboard">
  <header class="dashboard__header">...</header>
  <nav class="dashboard__sidebar">...</nav>
  <main class="dashboard__main">...</main>
  <aside class="dashboard__aside">...</aside>
</div>

Abnahmekriterium: Die Layoutbereiche sind semantisch nachvollziehbar und können im DOM eindeutig gefunden werden.

02 Grid-Spalten und Grid Areas aufbauen · Musterlösung: Grid Areas

Grid Areas machen das Layout lesbar. Namen in grid-template-areas müssen zu den grid-area-Zuweisungen passen.

.dashboard {
  display: grid;
  grid-template-columns: 220px minmax(0, 1fr) 280px;
  grid-template-areas:
    "header header header"
    "sidebar main aside";
  gap: 1rem;
}
.dashboard__header { grid-area: header; }
.dashboard__sidebar { grid-area: sidebar; }
.dashboard__main { grid-area: main; }
.dashboard__aside { grid-area: aside; }

Abnahmekriterium: Das Desktoplayout besitzt ein klares Raster; Bereiche werden nicht zufällig mit Abständen verschoben.

03 Responsives Kachelgrid mit auto-fit und minmax · Musterlösung: Kachelgrid

auto-fit und minmax erzeugen ein responsives Raster ohne viele einzelne Breakpoints. minmax(220px, 1fr) verhindert zu kleine Karten.

.tile-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(220px, 1fr));
  gap: 1rem;
}

Abnahmekriterium: Das Kachelgrid passt sich der verfügbaren Breite an und bleibt auch mit mehr Karten stabil.

04 Grid und Flexbox sinnvoll kombinieren · Musterlösung: Grid außen, Flex innen

Grid eignet sich für das zweidimensionale Seitenraster. Flexbox eignet sich innerhalb einer Card, wenn Inhalt auf einer Achse verteilt wird.

.status-card {
  display: flex;
  flex-direction: column;
  gap: .5rem;
}
.status-card__number {
  font-size: 2rem;
  font-weight: 700;
}

Abnahmekriterium: Die Lösung zeigt fachlich: Grid steuert das Raster, Flexbox richtet Inhalte innerhalb einzelner Komponenten aus.

05 Mobile Variante, Dokumentation und Code-Review · Musterlösung: mobile Grid-Variante

Mobil werden alle Bereiche in eine Spalte gebracht. Die Reihenfolge wird bewusst festgelegt, damit Zusatzinformationen nicht vor dem Hauptinhalt stehen.

@media (max-width: 760px) {
  .dashboard {
    grid-template-columns: 1fr;
    grid-template-areas:
      "header"
      "sidebar"
      "main"
      "aside";
  }
}

Abnahmekriterium: Die mobile Ansicht ist lesbar, die Reihenfolge ist fachlich sinnvoll und das Grid ist nachvollziehbar dokumentiert.

Lösungshinweise zu den Zusatzaufgaben Tag 3

Die vorhandenen Zusatzaufgaben sind nach Tag und Thema einsortiert. Wenn eine konkrete Lösungsdatei vorhanden ist, ist sie in der Aufgabenübersicht direkt verlinkt. Für Aufgaben ohne eindeutige Datei gelten diese fachlichen Lösungskriterien:

  • Tabellen semantisch mit table, tr, th, td strukturieren; Grid für Layout, Tabellen für Daten.
  • Positionierung nur dort nutzen, wo sie fachlich gebraucht wird.
  • Grid-Aufgaben müssen Spalten, Zeilen, gap und responsive Verhalten nachvollziehbar zeigen.
Tag 4 · Responsive Design · Media Queries · Medien · Lösungen

Responsive Design, Medien und Mobile First

Bestehende Tageslösung Tag 4 Teilnehmeraufgaben Tag 4

01 Mobile-First-Basis ohne Media Query · Musterlösung: Mobile First

Das Standard-CSS beschreibt zuerst die kleinste Ansicht. Größere Ansichten werden danach ergänzt. Dadurch bleiben kleine Geräte die Basis und keine Nachkorrektur.

.hero, .card-list, .site-nav {
  display: grid;
  gap: 1rem;
}
/* Noch keine Media Query: Das ist die mobile Basis. */

Abnahmekriterium: Ohne Media Query ist die Seite auf Smartphonebreite vollständig lesbar und bedienbar.

02 Container, flexible Abstände und Fluid Typography · Musterlösung: Container und clamp

max-width verhindert zu lange Zeilen. clamp() gibt Minimum, bevorzugten Wert und Maximum vor.

.container {
  width: min(100% - 2rem, 1120px);
  margin-inline: auto;
}
h1 {
  font-size: clamp(2rem, 5vw, 4rem);
}

Abnahmekriterium: Die Seite ist auf großen Bildschirmen nicht zu breit und skaliert Schrift/Abstände kontrolliert.

03 Breakpoints fachlich begründen · Musterlösung: Breakpoints

Ein Breakpoint wird dort gesetzt, wo Inhalt oder Navigation sichtbar nicht mehr gut funktionieren. Er ist kein Geräte-Name, sondern eine Layoutentscheidung.

@media (min-width: 768px) {
  .site-nav {
    display: flex;
    justify-content: space-between;
    align-items: center;
  }
  .hero {
    grid-template-columns: 1.1fr .9fr;
  }
}

Abnahmekriterium: Breakpoints sind nicht beliebig, sondern anhand konkreter Layoutprobleme begründet.

04 Bilder, Video und eingebettete Medien responsiv machen · Musterlösung: responsive Medien

Bilder und Videos bekommen eine maximale Breite und behalten ihre Proportionen. Eingebettete Medien sollten einen responsiven Wrapper oder aspect-ratio erhalten.

img, video, iframe {
  max-width: 100%;
}
img, video {
  height: auto;
}
.media-frame {
  aspect-ratio: 16 / 9;
  width: 100%;
}

Abnahmekriterium: Medien skalieren sauber, behalten sinnvolle Proportionen und bleiben auf Mobilgeräten bedienbar.

05 Responsive Abnahme und Fehlerkorrektur · Musterlösung: responsive Abnahme

Die Prüfung umfasst Viewports, horizontalen Scroll, Touchflächen, Lesbarkeit und tote CSS-Regeln. Fehler werden mit Ursache und Korrektur dokumentiert.

/* Checkliste */
/* 375px: kein horizontaler Scroll */
/* 768px: Navigation sinnvoll */
/* 1280px: Container begrenzt */
/* Buttons: ausreichend groß und fokussierbar */

Abnahmekriterium: Die Seite ist bei den getesteten Viewports stabil; Fehler wurden dokumentiert und behoben.

Lösungshinweise zu den Zusatzaufgaben Tag 4

Die vorhandenen Zusatzaufgaben sind nach Tag und Thema einsortiert. Wenn eine konkrete Lösungsdatei vorhanden ist, ist sie in der Aufgabenübersicht direkt verlinkt. Für Aufgaben ohne eindeutige Datei gelten diese fachlichen Lösungskriterien:

  • Bilder, Videos und eingebettete Medien müssen responsive begrenzt werden.
  • alt-Texte und controls sind bei Bild/Video-Aufgaben Pflicht, wenn fachlich passend.
  • Media Queries werden anhand realer Layoutbrüche gesetzt und bei 375px, 768px und 1280px geprüft.
Tag 5 · Webfonts · Refactoring · Abschluss · Lösungen

Webfonts, Formulare, Refactoring und Abschlussprojekt

Bestehende Tageslösung Tag 5 Teilnehmeraufgaben Tag 5

01 Lokale Webfonts datenschutzfreundlich einbinden · Musterlösung: lokale Webfonts

Lokale Webfonts vermeiden unnötige externe Requests. Die Lizenz wird dokumentiert; font-display: swap verhindert unsichtbaren Text beim Laden.

@font-face {
  font-family: "CourseSans";
  src: url("../fonts/CourseSans.woff2") format("woff2");
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}
body {
  font-family: "CourseSans", Arial, Helvetica, sans-serif;
}

Abnahmekriterium: Die Schrift wird lokal geladen, es gibt eine Lizenznotiz und bei Ladeproblemen existiert ein brauchbarer Fallback.

02 Formular erstellen und nach UI/UX-Grundlagen gestalten · Musterlösung: Formular

Jede Eingabe erhält ein Label. Fokuszustände und Abstände zeigen den aktuellen Bedienkontext. Mobil bleibt das Formular einspaltig.

<form class="contact-form">
  <label for="name">Name</label>
  <input id="name" name="name" type="text" required>
  <label for="message">Nachricht</label>
  <textarea id="message" name="message"></textarea>
  <button class="button" type="submit">Absenden</button>
</form>

Abnahmekriterium: Das Formular ist semantisch, visuell verständlich, per Tastatur bedienbar und mobil nutzbar.

03 CSS refactoren und wiederholte Muster ersetzen · Musterlösung: Refactoring

Ziel ist nicht weniger CSS um jeden Preis, sondern besser wartbares CSS: zentrale Werte, wiederverwendbare Komponenten und klare Klassennamen.

:root { --radius-card: 1rem; --shadow-card: 0 12px 28px rgba(0,57,100,.08); }
.card {
  border-radius: var(--radius-card);
  box-shadow: var(--shadow-card);
}

Abnahmekriterium: Die CSS-Datei ist wartbarer, Werte sind zentralisiert und Selektoren bleiben verständlich.

04 Moderne CSS-Funktionen und Effekte gezielt einsetzen · Musterlösung: calc, clamp und Übergänge

calc() kombiniert Werte unterschiedlicher Einheiten. clamp() begrenzt Skalierung. Transition/Transform sollten dezent bleiben.

.feature {
  width: calc(100% - 2rem);
  padding: clamp(1rem, 3vw, 2rem);
  transition: transform .2s ease;
}
.feature:hover { transform: translateY(-2px); }

Abnahmekriterium: Moderne CSS-Funktionen werden sinnvoll eingesetzt und nicht nur dekorativ eingebaut.

05 Abschlussprojekt prüfen, präsentieren und abgeben · Musterlösung: Abschlussabgabe

Die Abgabe enthält lauffähige HTML/CSS-Dateien, Assets, Font-Lizenznotiz, kurze technische Beschreibung und eine Reflexion. Wichtig ist, dass alle Pfade nach dem Entpacken funktionieren.

abgabe.zip
├─ index.html
├─ css/style.css
├─ images/...
├─ fonts/README_Fonts.txt
└─ kurzbeschreibung.txt

Abnahmekriterium: Die Abgabe ist vollständig, lauffähig, fachlich erklärbar und enthält eine kurze Reflexion.

Lösungshinweise zu den Zusatzaufgaben Tag 5

Die vorhandenen Zusatzaufgaben sind nach Tag und Thema einsortiert. Wenn eine konkrete Lösungsdatei vorhanden ist, ist sie in der Aufgabenübersicht direkt verlinkt. Für Aufgaben ohne eindeutige Datei gelten diese fachlichen Lösungskriterien:

  • Formulare müssen mit Labels, sinnvollen Eingabetypen und sichtbarem Fokus aufgebaut werden.
  • Webfonts brauchen Lizenz-/Quellenhinweis und Fallbacks.
  • Refactoring zeigt weniger Wiederholung, klarere Komponenten und zentral gepflegte Werte.
