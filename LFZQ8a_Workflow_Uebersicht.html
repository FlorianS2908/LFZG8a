Dozenten-Teilnehmer-System

LFZQ8a Workshop-App verstehen und starten

Dieses Dokument erklaert den geplanten Workflow der Electron-App: Dozenten steuern Freigaben, sehen Arbeitsfortschritte und nutzen die Kursmaterialien in einer integrierten App-Oberflaeche. Teilnehmer arbeiten lokal mit Projektdateien und melden ihren Status zurueck.

Zielbild

Die Anwendung soll Unterrichtsmaterial, Freigaben, Projektarbeit und Fortschrittsuebersicht in einem gefuehrten Workshop-System zusammenbringen.

Zentrale Steuerung

Integrierte Dozentenview

Der Dozent startet die Electron-App, oeffnet Materialien im App-Viewer, gibt Bereiche frei und sieht eine Teilnehmeruebersicht.

Teilnehmerzugang

Teilnehmer-View

Teilnehmer sehen nur freigegebene Tage, Tools und Projekte. Beim Erststart erfassen sie ihr Profil.

Arbeitsdateien

Lokale Ausgangssituation

Projektdateien liegen lokal beim Teilnehmer, damit sie in VS Code bearbeitet werden koennen.

Aufraeumziel

Nur Arbeitsdateien standalone

Tage, Tools, Aufgabenpakete und Projektseiten werden ueber die App gefuehrt. Eigenstaendig bleiben nur die Dateien, an denen wirklich gearbeitet wird.

Rollen

Dozenten- und Teilnehmerbereich bleiben getrennt, arbeiten aber ueber den Kursserver zusammen.

Rolle Aufgabe Sichtbare Inhalte Besonderheit
Dozent Kurs steuern, Materialien oeffnen, Teilnehmer begleiten Tage, Tools, Projekte, Loesungen, Freigaben, Teilnehmerstatus Startet die Electron-App und den lokalen Kursserver
Teilnehmer Aufgaben bearbeiten, Status melden, Hilfe anfordern Nur freigegebene Tage, Tools und Projekte Legt beim Erststart ein Profil an
Kursserver Freigaben, Profile und Fortschritte austauschen API und Teilnehmer-Weboberflaeche Laeuft lokal auf dem Dozenten-Client

Branch und Entwicklungsstand

Die aktuelle Umsetzung liegt auf dem Electron-Desktop-Zweig des Projekts.

Aktueller Branch

electronDesktop

Auf diesem Branch wird die Desktop-App mit integrierter Dozentenview, Wizard, Kursserver, Teilnehmerfreigaben und Fortschrittsuebersicht entwickelt.

Abgrenzung

DSGVO-Diagnose getrennt

Diagnosefunktionen mit technischen Geraete- oder Netzwerkdaten liegen nicht in diesem Arbeitsstand, sondern wurden separat ausgelagert.

Arbeitsstand

Lokaler Kursbetrieb mit App-Viewer

Der Stand ist fuer lokale Tests gedacht: Dozent startet Electron, Inhalte werden ueber den Kurskatalog in der App angezeigt, Teilnehmer verbinden sich ueber die angezeigte Kursserver-Adresse.

Modularitaet

Tage, Projekte und Tools getrennt

Die App nutzt getrennte Katalogmodule unter desktop-app/app/lib/catalog/. Kurstage, Projekte, Tools/Leitfaeden und Teilnehmerinhalte koennen dadurch getrennt gepflegt werden, waehrend alles weiterhin in der Electron-App angezeigt wird.

Software und Voraussetzungen

Die App soll moeglichst per Doppelklick starten. Fehlende Abhaengigkeiten werden, soweit moeglich, automatisch eingerichtet.

Fuer die Paketbereitstellung wurde keine neue Laufzeit eingefuehrt. Die Starter pruefen benoetigte Software automatisch und nutzen, wenn moeglich, winget. Eine reine Paketliste liegt zusaetzlich in SOFTWARE.md.
Fuer Wartbarkeit und Erweiterbarkeit wurde ebenfalls keine neue Software eingefuehrt. Die Trennung erfolgt ueber JavaScript-Module: teacher-days.js, teacher-projects.js, teacher-tools.js, participant-content.js und die Renderer-Gruppierung course-content-groups.js.
Komponente Wird benoetigt fuer Installation / Bereitstellung
Windows Dozenten- und Teilnehmer-Clients im Kursraum Voraussetzung fuer das aktuelle Startskript und die geplante Desktop-Nutzung
Node.js LTS Start der Electron-App und Installation der App-Abhaengigkeiten Wenn Node.js fehlt, versucht das Startskript die Installation ueber winget
winget Automatische Node.js-Installation auf Windows Falls weder Node.js noch winget vorhanden ist, muss Node.js LTS manuell installiert werden
Electron Dozenten-Desktop-App, Wizard, integrierte Dozentenview und Kursserver Wird ueber pnpm install oder npm install aus desktop-app/package.json installiert
pnpm oder npm Installation der JavaScript-Abhaengigkeiten Das Startskript nutzt bevorzugt pnpm, sonst npm
Browser Teilnehmer-View ueber die Kursserver-Adresse Teilnehmer oeffnen die angezeigte URL im Browser
Visual Studio Code Bearbeitung der lokalen Ausgangssituationen Empfohlen fuer Teilnehmer und Dozent; Arbeitsordner werden aus der Electron-App direkt geoeffnet
Fuer einen spaeteren Rollout ist ein fertiger Installer oder ZIP-Container sinnvoll. Dann muessen Teilnehmer nicht manuell im Projektordner arbeiten, sondern starten ein vorbereitetes Paket mit Arbeitsdateien und Verbindung zum Dozentenserver.

Workflow

Der Unterricht folgt einem einfachen Ablauf von Einrichtung ueber Freigabe bis zur Fortschrittskontrolle.

1. Dozent startetElectron-App oeffnen, Wizard bei Bedarf abschliessen; danach liegen Settings im Profil-Icon. Bei aktivem zweiten Monitor erscheint die Freigabe-View automatisch separat.
2. Kursserver laeuftDie Dozentenview zeigt die Teilnehmeradresse an.
3. Teilnehmer melden sichProfil mit Name, Kuerzel, Profilbild und Sprache anlegen oder ueber das Profil-Icon bearbeiten.
4. FreigebenDozent gibt Tage, Projekte und Tools fuer alle frei.
5. ArbeitenTeilnehmer bearbeiten lokale Ausgangssituationen.
6. BegleitenDozent sieht Fortschritt, Hilfeanfragen und letzte Aktivitaet.

Die eingebauten Aufgabenpakete werden ueber eine zentrale Registry in der Electron-App verwaltet. Dadurch kann der Dozent Aufgaben und Loesungen einzeln freigeben, ohne direkte Dateipfade im Teilnehmerbereich offenzulegen.

Teststrategie

Die automatisierten Tests sichern App-Logik, Pfade, Rollen-Trennung und Bereitstellung mit einem festen Coverage-Ziel ab.

Coverage

95 Prozent als Mindestwert

Der Test-Runner misst V8/Node-Coverage fuer desktop-app/app/lib und bricht unter 95 Prozent Pfadabdeckung ab.

Blackbox

Oeffentliche Funktionen

Profile, Freigaben, Fortschritt, Reports, Pfadregeln und Deployment-Dateien werden ueber Eingaben und erwartete Ausgaben geprueft.

Whitebox

Interne Randfaelle

Fallbacks, Fehlerpfade, kaputte JSON-Dateien, Display-Auswahl, Fortschrittsgrenzen und deaktivierte Historie sind gezielt abgedeckt.

C0 bis C4 im Projekt

Stufe Abdeckung Umsetzung
C0 Anweisungen Zentrale Daten-, Display- und JSON-Funktionen werden direkt ausgefuehrt.
C1 Zweige Erfolgs-, Fallback-, Fehler- und Grenzwertzweige werden getestet.
C2 Bedingungen Optionen wie Historie, Netzwerkdaten, Freigaben und Hilfe-Status laufen in beiden Richtungen.
C3 Pfade Wichtige App-Flows fuer Speicherung, Profile, Fortschritt, Reports und Deployment sind automatisiert.
C4 Mehrfachbedingungen Kombinierte Randfaelle wie fehlende Eingaben plus vorhandene Daten oder unvollstaendige Netzwerkdaten plus HTML-Report sind abgedeckt.

Lokal testen

Der Workflow kann lokal simuliert werden, ohne mehrere echte Clients zu verwenden.

Dozent

App starten

Dozenten-App per Startskript oeffnen und in der Teilnehmeruebersicht die lokale Teilnehmeradresse ablesen.

Teilnehmer

Browserfenster nutzen

Die Teilnehmeradresse in einem Browser, Inkognito-Fenster oder zweiten Browserprofil oeffnen.

Interaktion

Freigabe und Fortschritt pruefen

Profil anlegen, Statusbuttons klicken und pruefen, ob die Dozentenuebersicht aktualisiert wird.

Konkreter Testablauf

  1. ContentFactory per Doppelklick auf ContentFactoryMainStart.cmd starten.
  2. In der Dozentenview den Bereich Teilnehmer oeffnen.
  3. Dort die angezeigte Teilnehmer-Adresse ablesen, zum Beispiel http://localhost:PORT/teilnehmer.
  4. Diese Adresse in einem normalen Browserfenster oeffnen. Dieses Fenster simuliert den ersten Teilnehmer.
  5. Beim ersten Oeffnen das Teilnehmerprofil ausfuellen, zum Beispiel Name Test Teilnehmer und Kuerzel TT. E-Mail und Teams-Name sind optional.
  6. Nach dem Speichern sollte der Teilnehmer nach wenigen Sekunden in der Dozentenview sichtbar werden.
  7. Im Teilnehmerfenster bei einem freigegebenen Bereich die Statusbuttons testen: Gestartet, In Bearbeitung, Hilfe benoetigt und Erledigt.
  8. In der Dozentenview pruefen, ob aktuelle Aufgabe, Fortschritt, Hilfe-Status und letzte Aktivitaet aktualisiert werden.
Freigaben testen

Bereiche entsperren

  • In der Dozentenview den Bereich Freigaben oeffnen.
  • Zum Beispiel Tag 2 oder Projektmaterialien aktivieren.
  • Freigaben speichern klicken.
  • Im Teilnehmerfenster kurz warten oder neu laden.
  • Der vorher gesperrte Bereich sollte danach freigegeben und anklickbar sein.
Mehrere Teilnehmer

Mehrere Profile simulieren

  • Teilnehmer-Adresse in einem Inkognito-Fenster oeffnen.
  • Alternativ ein zweites Browserprofil verwenden.
  • Alternativ einen anderen Browser verwenden.
  • Je Fenster ein eigenes Profil anlegen, damit mehrere Teilnehmer in der Dozentenuebersicht erscheinen.
